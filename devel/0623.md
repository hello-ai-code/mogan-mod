# [0623] 支持在 HTML 导出时将所有生成的图片均以 Base64 形式内联嵌入，实现单网页自包含导出

当 Mogan 导出包含图片的文档为 HTML 时，默认会将生成的公式和图片存储为外部独立的 `.png` 文件（如 `image-1.png` 等）。这种多文件结构对于网络分发、邮件发送及团队协作非常不便。

本项目实现了在 HTML 导出时，全自动将所有生成的过渡性图片以 Base64 Data URL 形式直接内联嵌入 HTML 页面中，且在导出后自动清空本地一切过渡图片，从而实现纯净的单文件自包含 HTML 导出。同时，该项目彻底优化了对内联/嵌入式元组图片数据的解析与无损转换。

## 5 What
1. **支持 Base64 内联图片嵌入 HTML**：引入并实现了 `texmacs->html:base64` 首选项控制（默认开启），通过调用 Goldfish 内置 of `(liii base64)` 库算法，对生成的 PNG 文件内容进行二进制安全的 Base64 编码，并以内联 Data URL 的形式直接内嵌在 `<img>` 的 `src` 属性中。
2. **零痕迹文件清理**：在 Base64 模式下导出后，自动清空留在临时目录和目标工作目录下的一切过渡型外部图片文件（`.pdf`、`.png`），完美实现“零浪费”清洁导出。
3. **支持多种嵌入式图片格式解析**：在 `tmhtml.scm` 中全新实现了通用的内联图片提取函数 `tmhtml-extract-embedded`。不论是带有 `raw-data` 节点包装的传统 tuple 格式，还是直接在 tuple 根部放置 raw bytes 且无 wrapper 的简洁 tuple 格式，均可无损读取其对应的二进制内容并落地转换，彻底修复了嵌入式图片导出失败的系统缺陷。
4. **支持 GUI 专属的 Qt 弹窗进度条，保持命令行静默，并引入百分比插值平滑动画**：在 HTML 导出过程中，为 GUI 环境增加基于自定义 `QDialog` 的弹窗进度条。该设计通过 QSS 样式表锁定外框颜色（`border: 1px solid grey;`），并引入 50ms 线程休眠（`QThread::msleep(50)`）以确保 Qt 事件循环有充足时间完成首帧及后续进度的重绘，彻底解决界面假死与重绘不全的缺陷。同时，进度条范围锁定为 `0-100`，并引入了基于 `(target - prev) / 10` 的百分比插值平滑动画机制，通过 15ms 的微休眠（`QThread::msleep(15)`）和强制重绘，将原本离散、生硬突跳的进度更新重构为 10% 级极佳细粒度的平滑滑行动画，彻底解决视觉生硬问题。此外，进行了强隔离设计，若在命令行或 headless 状态下调用 tmu -> html 导出时，保持绝对静默与安全，不展示任何进度条或弹窗。
5. **引入专属进度计数器 `tmhtml-image-progress`**：将原本用于生成图片文件名的 `tmhtml-image-serial` 序号与进度条更新逻辑彻底解耦。由于多阶段图片生成或缓存机制可能导致 `tmhtml-image-serial` 出现非线性递增或跳跃，直接使用其作为进度值会导致进度条中途跳满或比例失调。通过引入专属、精确递增的 `tmhtml-image-progress` 计数器，在每次实际导出/渲染图片时精确加 1，从而保证进度条的指示与实际导出进度 100% 吻合。

## 6 Why
* **单网页分享需求**：默认导出多幅外部 `.png` 会极大地削弱科学文档导出的可分享性，极易发生图片丢失、链接失效等情况。通过 Base64 内联嵌入，可以让导出的网页百分之百自包含，便于携带与直接传播。
* **解析失败原因**：传统的内联图片逻辑严格判定 `(func? (car l) 'tuple 2)` 且 `(func? (cadar l) 'raw-data 1)`。然而有些嵌入式图片会直接包含 hex bytevector 数据（即 tuple 的第一个元素直接是 bytevector/string 而不是 `raw-data` 结点）。从而导致 `tmhtml-image` 回退至 `nstring?` 判定，跳过了将其保存至临时磁盘文件的步骤，并在后续引发文件找不到和尺寸读取的错误。

## 7 How
1. 设计通用的嵌入式数据提取器 `tmhtml-extract-embedded`，兼容多种嵌入格式，支持无损读取 tuple 里的 raw bytes 并安全转换为 `byte-vector` 供外部转换器（如 MuPDF 或 PDF Hummus）读取。
2. 实现 R7RS 二进制安全的读取器 `tmhtml-read-binary-file` 与写入器 `tmhtml-write-binary-file`，绕过 C++ `string-load` 的截断和 `display` 的 UTF-8 编码转换，保证文件无损读写。
3. 通过首选项配置 `texmacs->html:base64`（默认开启）进行逻辑切分：
   - 开启时：将转换后的 PNG 读为字节数组并由 `(liii base64)` 的 `bytevector-base64-encode` 编码后，采用 `data:image/png;base64,...` 的形式直接赋值给 `<img>`。随后自动调用 `url-remove` 将临时生成的 `.pdf` 及 `.png` 文件从磁盘中一并彻底抹除。
   - 关闭时：恢复经典的外部图片文件名引用模式，不删除外部图片文件。
   - **引入专属进度计数器**：在 `tmhtml.scm` 中定义并初始化 `tmhtml-image-progress` 为 0。在 `tmhtml-initialize` 中将其重置为 0。在 `tmhtml-png` 实际执行图片导出与渲染时，执行 `(set! tmhtml-image-progress (+ tmhtml-image-progress 1))`，并调用 `(html-progress-update tmhtml-image-progress)`。这成功将图片命名序列 `tmhtml-image-serial`（在 `tmhtml-image-names` 中递增，易受多阶段调用或缓存影响而产生跳跃）与实际进度解耦，彻底解决了进度条中途跳满或比例不准的逻辑缺陷。
4. 在 `src/Plugins/Html/` 目录（C++ 侧）及 `src/Scheme/Plugins/glue_html.lua`（胶水层）中全新封装了基于自定义 `QDialog` 的弹窗进度条。在 Scheme 侧，只有当 `(qt-gui?)` 为真且确实正在真正的 `QApplication` 主事件循环环境下（使用 `QApplication::instance()` 检查）才实例化弹窗，从而完全隔离和保障了命令行的静默无弹窗运行安全。在 C++ 侧，自定义 `QDialog` 采用 `QVBoxLayout` 与 `QHBoxLayout` 布局，包含 `QLabel`、`QProgressBar` 和 `QPushButton`（取消按钮），通过 QSS 样式表锁定外框颜色（`border: 1px solid grey;`），并在 `html_progress_start` 中引入 `QThread::msleep(50)` 线程休眠，配合 `QCoreApplication::processEvents()` 强制刷新，彻底解决了 macOS 等平台下进度条重绘假死的缺陷。在 `html_progress_update` 中，将进度条范围锁定为 `0-100`，并实现 `(target - prev) / 10` 的插值平滑动画机制：当目标进度 `target` 大于当前进度 `prev` 时，计算步长 `step = (target - prev) / 10`（最小为 1），通过循环逐步递增进度值，每次递增后调用 `bar->repaint()`、`QCoreApplication::processEvents()` 并配合 `QThread::msleep(15)` 微休眠，从而将原本离散、生硬突跳的进度更新重构为 10% 级极佳细粒度的平滑滑行动画，彻底解决视觉生硬问题。
