# [200_60] 修复自动备份中嵌入图片损坏的问题

## 如何测试（用户场景）

### 场景一：嵌入图片的文档可以从自动备份中恢复

1. 打开 Mogan，新建一个文档
2. 通过菜单 `Insert -> Image -> Insert image` 插入一张本地图片
   - 注意选择 `Insert image`，不要选择 `Link image`
   - 这样图片会以内嵌数据的形式保存到文档中
3. 将文档保存为 `.tm` 或 `.tmu`
4. 修改文档内容，例如输入一行文字
5. 触发自动备份
   - 可以等待自动备份定时触发
   - 也可以在设置中点击自动备份入口手动打开备份目录，触发一次备份
6. 打开 `$TEXMACS_HOME_PATH/system/backup/`
7. 找到最新生成的 `.tmu` 备份文件并打开
8. 预期结果：
   - 文档可以正常打开
   - 内嵌图片可以正常显示
   - 图片不是空白、损坏图标或无法解析的内容

### 场景二：自动备份不会静默改写原始文档

1. 打开一个已有文档，确保文档中原本没有 `stem-doc-id`
2. 触发自动备份，但不要手动保存原文档
3. 检查原文档文件内容
4. 预期结果：
   - 原文档不会因为自动备份被写回
   - 新生成的备份 `.tmu` 中包含用于 manifest 追踪的 `stem-doc-id`

## 2026/04/30 自动备份统一使用 tmu 格式并复用 buffer-export

### What

修复自动备份生成的 `.tmu` 文件中，嵌入图片可能损坏的问题。

自动备份仍然统一存储为 `.tmu` 快照文件，但实际导出逻辑改为复用手动保存和自动保存使用的 `buffer-export` 路径。

### Why

手动保存和自动保存中的图片没有损坏，是因为二者都走 C++ 的 `buffer_export` 链路：

```scheme
(buffer-save name)
(buffer-export name aname fm)
```

该路径会从 editor/view 中同步最新 `new_data`，再通过 `attach_data` 组装完整文档，最后序列化写入文件。嵌入图片的 `RAW_DATA` 在这一过程中保持二进制数据，不会经过 Scheme stree/list 形态的转换。

旧的自动备份逻辑不同：

```scheme
(buffer-get name)
(auto-backup-doc-with-doc-id doc doc-id)
(tree-export (tm->tree doc*) target "tmu")
```

其中 `auto-backup-doc-with-doc-id` 会通过 `tmfile-extract` / `tmfile-assign` 修改文档的 `initial` collection。这些辅助函数会把 tree 转成 stree/list 形态，而 stree 中的 `raw-data` 会被编码为 base64。

问题在于最后使用的是 `(tm->tree doc*)`。`tm->tree` 只按普通内容构造 tree，不会把 stree 中的 base64 `raw-data` 解回原始二进制，导致备份文件中的图片数据变成 base64 文本的字节，从而损坏。

### 涉及文件

- `TeXmacs/progs/texmacs/texmacs/tm-files.scm`
  - 修改 `auto-backup-export-buffer`
  - 不再使用 `tree-export (tm->tree doc*)`
  - 改为写入当前 buffer 的 `stem-doc-id` 后调用 `buffer-export`

### 实现说明

新的自动备份导出逻辑：

```scheme
(tm-define (auto-backup-export-buffer name target info)
  (catch #t
    (lambda ()
      (let ((doc-id (assoc-ref info "doc_id")))
        (with-buffer name
          (init-env "stem-doc-id" doc-id))
        (if (buffer-export name (auto-backup-path->url target) "tmu")
            #f
            (auto-backup-file-size target))))
    ...))
```

这里仍然固定传入 `"tmu"`，因此自动备份文件继续统一为 `.tmu` 格式。

`stem-doc-id` 通过 `init-env` 写入当前 buffer 环境。随后 `buffer-export` 会同步 editor data 并通过 `attach_data` 生成完整导出文档，所以备份文件中会包含该 doc id。

### 与自动保存的关系

自动保存会沿用原文档格式：

```scheme
(let* ((aname (url-autosave name suffix))
       (fm (url-format name)))
  (buffer-export name aname fm))
```

自动备份与自动保存的定位不同：

- 自动保存用于崩溃恢复，适合保留原文档格式
- 自动备份用于版本快照和 manifest 管理，统一 `.tmu` 更便于索引、恢复和后续上传

因此本次修复只复用 `buffer-export` 的安全导出链路，不复用自动保存的目标文件命名和格式选择策略。

### 行为变化

- 自动备份文件仍生成在 `$TEXMACS_HOME_PATH/system/backup/`
- 自动备份文件后缀仍为 `.tmu`
- manifest 结构、md5 去重、版本保留策略不变
- 自动备份不会静默改写原始文档
- 如果原文档缺少 `stem-doc-id`，自动备份文件会包含当前会话绑定的 doc id；用户手动保存后，原文档才会持久化该 doc id
