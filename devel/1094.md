## 3 如何测试

### 3.1 确定性测试（单元测试）
本任务为工具脚本 + CI 配置改动，无对应 C++/Scheme 单元测试。

### 3.2 非确定性测试（脚本验证）
```bash
# 1. 干净工作树：--check 应通过、退出码 0、不修改任何文件
gf format --check ; echo "exit: $?"          # 期望 exit 0
git status --short                            # 期望无新增改动

# 2. 故意改坏一个 scm-dirs 下的 .scm：应报错并退出码 1
printf '\n\n\n' >> TeXmacs/progs/utils/plugins/plugin-cmd.scm
gf format --check ; echo "exit: $?"          # 期望列出该文件、exit 1
git checkout TeXmacs/progs/utils/plugins/plugin-cmd.scm

# 3. 故意改坏一个 .cpp：应报错并退出码 1
echo "  int tmp;" >> tests/Data/view_history_test.cpp
gf format --check ; echo "exit: $?"          # 期望列出该文件、exit 1
git checkout tests/Data/view_history_test.cpp

# 4. 排除项不误报：prefix-kbd.scm 在 gfexclude.json 中，--check 不应标记它
gf format --check 2>&1 | grep -i prefix-kbd   # 期望无输出

# 5. 就地路径未回归（无 --check 仍就地格式化）
gf format                                     # 行为同改前
```

## 4 如何提交

提交前执行以下最少步骤：

```bash
gf fmt --changed-since=main
gf format --check        # 确认本次新增 .scm 已格式化、--check 幂等 exit 0
```

## 5 What
1. `goldformat-path.scm`：参照 `collect-cpp-files`/`collect-all-cpp-files` 新增 `collect-scm-files`/`collect-all-scm-files`，遍历 `scm-dirs` 收集 `.scm`。
2. `goldformat.scm`：给 `gf format` 增加 `--check` 子命令——非破坏性地检查 C++ 与 Scheme 是否已格式化，发现未格式化文件则退出码 1。就地格式化路径不变。
3. `ci-format-check-debian13.yml`：下载安装 goldfish `.deb`，新增 `gf format --check` 步骤，把 Scheme 格式也纳入 CI 门禁。

## 6 Why
`gf format` 此前只有「就地格式化」一种行为，且 `main` 不解析任何命令行参数——执行 `gf format --check` 会被静默忽略、**直接就地格式化整个工作树**，是个隐患。同时 CI（`ci-format-check-debian13.yml`）只用 clang-format 检 C++，**Scheme 格式无门禁**，依赖开发者自觉跑 `gf fmt`。

本任务：
- 给 `gf format` 加 `--check`：非破坏性检查，发现未格式化文件退出码 1，可作为 CI 门禁与本地 pre-commit 使用。
- 让 CI 同时检查 C++ 和 Scheme（`gf format --check` 一次覆盖），Scheme 格式从此也被自动把关。

**设计要点**（关键事实均已实测确认）：
- `main` 形参恒为 `()`，tool 参数经 `(scheme process-context)` 的 `command-line` 读取：`gf format --check` → `command-line` = `("goldfish" "format" "--check")`，用 `(member "--check" (cddr (command-line)))` 判断。
- C++ 检查：逐文件 `clang-format --dry-run --Werror`，`os-call` 返回退出码，非 0 即记入 offenders（stderr 重定向到 `/dev/null` 抑制 diff 噪声）。
- Scheme 检查：`gf fmt` 无退出码信号、`--dry-run` 只支持单文件，故逐文件 `sh -c "gf fmt --dry-run FILE > tmp"` 捕获输出，与磁盘内容 `string=?` 逐字节比较；对 `gfexclude.json` 排除文件（输出 `Skipped (excluded):`）视为通过。
- 退出码：`(exit N)` 经 `(scheme process-context)` 传播为进程退出码。
- Windows 无 `sh`，本地跑 `--check` 跳过并退出 0（不阻塞开发；CI 在 Debian 跑不受影响）。

## 7 How
1. `goldformat-path.scm`：加 `scm-exts '(".scm")`、`scm-file?`、`collect-scm-files`、`collect-all-scm-files`，并在 `(export ...)` 导出。
2. `goldformat.scm`：
   - import 增 `(liii string)`、`(scheme process-context)`。
   - 新增 `shell-quote`（单引号 POSIX 转义）、`excluded-output?`（识别 `Skipped (excluded):`）、`print-offenders`（打印各语言 offenders 清单）。
   - 新增 `check-cpp`（逐文件 clang-format dry-run）、`check-scm`（逐文件 gf fmt dry-run + 字节比较）、`run-check`（汇总、Windows 跳过、exit 1/0）。
   - 重写 `main`：有 `--check` 走 `run-check`，否则走原就地格式化路径。
3. `ci-format-check-debian13.yml`：paths 加 Scheme 相关路径；`Install dependencies` 增 `curl ca-certificates`（保留 `clang-format-19` 供 `gf format --check` 内部 C++ 检查）；新增「Download & install goldfish (latest)」——用 `releases/latest` 重定向动态解析最新 tag，下载 `goldfish-scheme-x86_64-<tag>.deb` 并 `dpkg -i`；用「Run gf format --check (C++ + Scheme)」单一步骤替换原有直接 clang-format 调用（在 repo 根跑，靠 cwd 的 `gfproject.json` 解析 mogan 的 `goldformat` 工具，C++ 检查复用同一套 `collect-all-cpp-files` 文件集）。
4. 验证：见第 3 节——本地干净树 exit 0、改坏 scm/cpp 各 exit 1 且列出文件、排除项不误报、就地路径不回归、CI job 跑通。

## 8 已知限制
- **`--check` 非严格只读**：`gf fmt --dry-run` 会填充 fmt 缓存（`~/.cache/goldfish/fmt/<version>/`），故 `--check` 会写缓存目录。缓存按内容哈希键控，不会导致误判，CI 临时 runner 无影响，本地无害。
- **CI 下载最新 goldfish**：deb 资产名含版本号（`goldfish-scheme-x86_64-<tag>.deb`），故用 `releases/latest` 重定向动态解析 tag 再拼资产名，不钉死版本。若将来 goldfish 改变 deb 命名规则（如 `x86_64` 改名），需同步更新 workflow 中的 glob/资产名。
