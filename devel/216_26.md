## 2026/04/22 新版json文件兼容旧scm文件

### What
本次补充了从旧版 `.scm` 存储向新版 `.json` 存储的兼容迁移层，覆盖三类历史数据：
- `interactive.scm` -> `interactive.json`
- `interactive.scm` 中的 `recent-buffer` -> `recent-files.json`
- `shortcuts.scm` -> `shortcuts.json`

与此前“仅在 JSON 为空时整体导入”的方案相比，这次改为更稳妥的按模块迁移：
- 每个模块各自维护一个 `scm->v1` 标记文件
- 若对应 marker 不存在，则尝试执行该模块的 SCM -> JSON 迁移
- 若 JSON 已存在，则不覆盖，而是将旧 SCM 中缺失的记录补充进去
- 若 JSON 不存在，则按当前空状态直接生成新的 JSON 文件

#### 修改文件

**TeXmacs/progs/kernel/texmacs/tm-dialogue.scm** (修改)
- 为 `interactive` 和 `recent-files` 分别增加独立迁移标记：
  - `$TEXMACS_HOME_PATH/system/interactive.scm->v1`
  - `$TEXMACS_HOME_PATH/system/recent-files.scm->v1`
- 保留旧版 `interactive.scm` 的读取能力，用于启动时一次性兼容迁移
- 新增旧格式解码辅助逻辑，将历史 `interactive.scm` 还原为可遍历的数据结构
- `interactive` 迁移改为“合并缺失记录”：
  - 读取旧版命令历史
  - 规范化后与现有 `interactive.json` 中同名命令的历史记录做合并
  - 完全相同的历史项不重复写入
- `recent-files` 迁移改为“按 path 补充缺失项”：
  - 从旧版 `recent-buffer` 中提取路径
  - 若 `recent-files.json` 中已存在同一路径，则跳过
  - 若不存在，则补充为新的 `files` 项，并补齐 `name / last_open / open_count / show`
- 调整 marker 写入时机：
  - 先保存 JSON
  - 再写入 `scm->v1` marker
  - 避免“marker 已写入但 JSON 尚未落盘”的不一致状态

**TeXmacs/progs/source/shortcut-edit.scm** (修改)
- 为自定义快捷键增加独立迁移标记：
  - `$TEXMACS_HOME_PATH/system/shortcuts.scm->v1`
- 保留旧版 `shortcuts.scm` 的读取能力，用于启动时兼容迁移
- `shortcuts` 迁移改为“按 shortcut 补充缺失项”：
  - 将旧版 `(shortcut command)` 结构转换为当前 JSON entry
  - 若 `shortcuts.json` 中已存在相同 `shortcut`，则跳过
  - 若不存在，则追加到现有快捷键列表
- 迁移结束后写回 `shortcuts.json`，并生成 `shortcuts.scm->v1` marker

### Why
1. 旧版数据最初存储在 `interactive.scm` 和 `shortcuts.scm` 中；后续迁移到 JSON 时，没有补上一层启动时兼容迁移，导致老用户升级后无法自动看到原有记录。
2. 仅用“JSON 是否为空”来判断是否迁移不够稳妥：
   - 用户可能已经有新的 JSON 文件，但其中只包含部分新版本产生的数据
   - 此时仍然需要把旧 SCM 中尚未迁移的记录补充进来，而不是直接放弃迁移
3. `interactive`、`recent-files`、`shortcuts` 三类数据的演进和使用场景不同，适合拆成三个独立模块管理各自迁移状态，而不是共用一个总开关。
4. 迁移后的正确行为应该是：
   - 不覆盖用户现有 JSON 数据
   - 只补充旧 SCM 中还未进入 JSON 的记录
   - 每个模块只迁移一次，迁移成功后以 `scm->v1` marker 作为状态确认

### How
1. **模块独立迁移**
   - `interactive`、`recent-files`、`shortcuts` 各自维护自己的 `scm->v1` 标记文件
   - 某个 marker 不存在时，只触发该模块的迁移逻辑
   - 这样即使某个模块已迁移完成，其他模块仍可单独补迁

2. **兼容读取旧 SCM**
   - `tm-dialogue.scm` 中重新引入旧版 `interactive.scm` 的解码逻辑
   - `shortcut-edit.scm` 中保留对旧版 `shortcuts.scm` 的读取
   - 这些兼容读取逻辑只用于迁移入口，不恢复旧运行时主路径

3. **补充合并而非覆盖**
   - `interactive`：
     - 先取出当前 `interactive.json` 中已有命令历史
     - 再将旧 `interactive.scm` 中同名命令的历史项做规范化
     - 用 `equal?` 判断是否为同一条历史项，只追加缺失项
   - `recent-files`：
     - 以 `path` 作为去重键
     - 旧 `recent-buffer` 中的路径只有在当前 JSON 中不存在时才追加
   - `shortcuts`：
     - 以 `shortcut` 作为去重键
     - 若当前 JSON 中已有同名快捷键，则旧 SCM 中对应记录不再导入

4. **Recent 顺序处理**
   - 旧 `recent-buffer` 只有路径，没有完整元数据
   - 迁移时为新增项补齐：
     - `name`: 由路径 basename 计算
     - `open_count`: 默认设为 `1`
     - `show`: 默认设为 `#t`
     - `last_open`: 基于当前 `recent-files.json` 中最小时间戳向前递减生成
   - 追加完成后再统一执行现有 LRU 排序逻辑，保持 recent 列表行为一致

5. **写盘顺序**
   - 迁移逻辑先将合并后的结果保存到 JSON
   - 保存成功后再写入 `scm->v1` marker
   - 后续启动时只要 marker 已存在，就不再重复执行该模块迁移
