## 3 如何测试

### 3.1 确定性测试（单元测试）

无新增单元测试。本 fix 位于 C++↔Scheme 集成边界，`ensureNewConversation` 依赖
Qt UI（`view_->createPanel`、`ChatConversationPanel` 信号槽），无法在 C++ 单元
测试层隔离验证。Scheme 侧 `chat-tab-session-set-thinking` 在 state 存在时一直
工作正常，无需新增测试。

### 3.2 非确定性测试（文档验证）

1. 启动 Mogan STEM，切换到 Chat Tab（自动创建一个空的新会话）
2. **在发送任何消息之前**，点击 reasoning（thinking）切换按钮将其启用
3. 发送一条消息
4. 确认消息 buffer 中出现 `<unfolded-explain>` 折叠块，展开后可见推理过程
   （或在 LLM 插件日志中确认请求 JSON 中携带了 `"thinking": "enabled"`）
5. 对照实验：在未应用本 fix 的版本上重复步骤 1–4，确认 reasoning 不生效，
   推理块不出现

## 5 What

在 `ChatController::ensureNewConversation` 创建新会话分支中，新增一次
`chat-persist-register-session` 调用，使 C++ 创建的新会话与 Scheme 侧会话
状态表保持同步：

```cpp
sessionManager_.setModel (sid, currentModel);
call ("chat-persist-register-session", sid, currentModel, string ("disabled"));
```

## 6 Why

**症状**：用户新建聊天页后、发送首条消息前启用 reasoning 模式无效。开启按钮
看似正常切换，但实际发送时 LLM 请求里 `"thinking"` 字段仍为 `"disabled"`。

**根因**：

`chat-tab-session-set-thinking`（`chat-tab-session.scm:84-95`）的实现：

```scheme
(tm-define (chat-tab-session-set-thinking session-id thinking)
  (let ((st (chat-tab-get-state session-id)))
    (when st
      (chat-tab-set-state! session-id
        (chat-tab-state (chat-tab-state-model st) thinking)))))
```

依赖 `chat-tab-get-state` 返回非 `#f` 的 state 对象。而 state 对象只有两条
创建路径：

| 路径 | 触发位置 | 是否正常 |
|------|---------|---------|
| `restoreSessionMeta`（恢复已有会话） | 调用 `chat-persist-register-session` | ✅ |
| `ensureNewConversation`（创建新会话） | **未** 调用 `chat-persist-register-session` | ❌ |
| 首次 `chat-tab-session-send` | 经由 `chat-tab-ensure-session!` 兜底 | 来不及 |

新会话直到用户首次发送消息时，才会由 `chat-tab-ensure-session!` 延迟创建
state。但 reasoning 切换按钮在发送**之前**就被点击——此时 state 尚不存在，
`when st` 保护使 set-thinking 静默 no-op，用户的设置被悄悄丢弃。

**对比**：恢复历史会话的 `chat-persist-load-all` 流程在 `chat-session-persist.scm:170`
显式调用了 `chat-persist-register-session`，所以恢复会话不受影响。新建会话
漏掉了这一步。

## 7 How

### 改动位置

`src/Plugins/Qt/qt_chat_controller.cpp` 中 `ensureNewConversation` 函数的
"无可复用会话，创建新会话" 分支（约第 568–577 行）。

### 改动内容

在 `sessionManager_.setModel(sid, currentModel)` 之后、`chat-tab-sync-dark-style!`
之前，插入一次 `chat-persist-register-session` 调用，参数与 `restoreSessionMeta`
保持一致：

```cpp
string                 sid  = sessionManager_.createSession ();
ChatConversationPanel* panel= view_->createPanel (sid);
if (!panel) return;

sessionManager_.setPanel (sid, panel);
sessionManager_.setModel (sid, currentModel);
call ("chat-persist-register-session", sid, currentModel, string ("disabled"));

call ("chat-tab-sync-dark-style!", sid);
```

### 安全性分析

- **幂等**：`chat-persist-register-session` 内部有 `(when (not st) ...)`
  保护，已存在 state 时直接跳过，不会覆盖。即使后续 `chat-tab-ensure-session!`
  被再次触发也不会冲突。
- **不影响恢复流程**：`restoreSessionMeta` 路径独立调用，行为不变。
- **时序正确**：注册发生在 panel 创建之后、暗色样式同步和 UI 信号连接之前，
  确保 reasoning 切换按钮在 UI 可见时 state 已就绪。

### 为什么不修复 Scheme 侧的 `set-thinking` 让它兜底创建 state？

`set-thinking` 时缺少 model 信息（model 由 C++ 通过 `setModel` 维护，Scheme
无法仅从 session-id 反查）。强制在 `set-thinking` 中创建空 state 会引入
"无 model 的伪 state"，污染后续 `chat-tab-state->plugin-session-id` 的拼接。
根治点应在"会话创建"侧而非"状态修改"侧。
