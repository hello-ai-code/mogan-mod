# [221_7] 修复右键菜单 “复制到” 的 crash 问题

## 如何测试

新建空文档，打开右键菜单，鼠标挪到 “复制到”，软件不应崩溃

## 2026/04/29 修复右键菜单 “复制到” 的 crash 问题

### Why

右键菜单里的 “复制到” 子菜单定义在 `main-menu.scm`：

```scheme
(tm-menu (texmacs-popup-menu)
  ("Smart paste" (kbd-magic-paste))
  ("Paste special" (interactive-paste-special))
  (=> "Copy to" (link clipboard-copy-export-menu))
  ---
  (former))
```

但是 `clipboard-copy-export-menu` 这个符号并不在 `main-menu.scm` 里定义，而是在 `edit-menu.scm` 里定义：

```scheme
(tm-define (clipboard-copy-export-menu)
  (clipboard-extern-menu converters-from-special clipboard-copy-export))
```

原来的问题是 `main-menu.scm` 直接使用了 `clipboard-copy-export-menu`，却没有显式 `:use (texmacs menus edit-menu)`。这样一来，这个符号是否存在，就取决于 `edit-menu.scm` 有没有在更早的时候“碰巧”被加载。

在 macOS 上，这个问题更容易稳定复现。右键事件会直接进入 popup 菜单路径，由 `edit_mouse.cpp` 构造 `texmacs-popup-menu`。如果这时 `edit-menu.scm` 还没有被装载，那么当鼠标移到 “复制到” 并展开子菜单时，Guile 就会在求值 `(link clipboard-copy-export-menu)` 时抛出：

```scheme
(unbound-variable ... (clipboard-copy-export-menu))
```

这个异常没有在菜单展开路径上被消化，最终表现为 crash。

Linux 和 Windows 上通常不容易复现，不是因为代码没有问题，而是因为主菜单/编辑菜单往往会更早加载，`clipboard-copy-export-menu` 已经提前进入环境，所以同一个依赖遗漏被掩盖了。

修复方式就是在 `main-menu.scm` 里显式声明对 `edit-menu` 的依赖，消除对加载顺序的隐式假设。
