# 216_21 模板中心远程更新后自动清除本地缓存

## 如何测试
1. 编译：`xmake b stem`
2. 联网打开启动页 `Template Center`，下载一个模板（点击 `Use Template`）。
3. 退出应用，模拟远程模板更新：
   - 编辑模板元数据缓存 `metadata.json`
   - 找到刚下载的模板条目，将其 `updated_at` 时间修改为比当前时间更晚的时间
   - `metadata.json` 位置：`$TEXMACS_HOME_PATH/system/template_cache/metadata.json`
4. 重新启动应用并联网打开 `Template Center`。
5. 点击刚才修改过 `updated_at` 的模板：
   - 预期行为：弹出下载进度条，重新下载模板后打开
   - 修复前行为：直接打开本地缓存的旧版本，没有下载过程
6. 检查本地缓存目录，确认旧版本模板文件已被删除：
   - 缓存目录：`$TEXMACS_HOME_PATH/system/template_cache/templates/`

## 2026/04/17 实现说明

### What
修复模板中心缓存逻辑：当远程模板更新后，自动清除本地缓存，确保用户获取到最新版本。

#### 修改文件

**src/Mogan/TemplateCenter/template_manager.cpp**

1. `isTemplateAvailableLocally()`: 改为直接检查文件是否存在，而非依赖可能过期的 `isLocal` 标志
   - 原逻辑：`return tmpl->isLocal || cache_->isTemplateCached(templateId);`
   - 新逻辑：先检查 `tmpl->localPath` 文件是否存在，不存在再检查缓存索引

2. `mergeMetadata()`: 
   - 增加远程更新时间检测逻辑，当 `remoteTmpl->updatedAt > existing->updatedAt` 且本地有缓存时：
     - 调用 `cache_->removeCachedTemplate(id)` 删除旧版本文件
     - 清除 `localPath` 和 `isLocal` 标记
     - 用户下次点击时触发重新下载
   - 添加 MD5 校验作为备用更新检测机制（当时间戳相同时，如果 MD5 变化也视为更新）
   - 修复末尾循环逻辑：只在 `!tmpl->isLocal` 时才设置缓存状态，防止刚清除的模板被重新标记

**src/Mogan/TemplateCenter/template_cache.cpp**
- `removeCachedTemplate()`: 添加文件删除结果检查和调试日志
- `loadMetadataCache()`: 只在文件实际存在时才设置 `isLocal=true`，防止缓存索引和实际文件状态不一致

### Why
原来的问题是：
- 用户下载模板后，模板文件被缓存到本地。
- 远程模板更新后，`mergeMetadata()` 只更新了元数据字段。
- 用户点击模板时，`isTemplateAvailableLocally()` 返回 true，直接打开了旧版本。
- 用户无法获取到远程更新的模板内容。

还发现一个边界情况：
- 清除缓存后，如果下载被中断或文件删除失败，`isLocal` 标志可能被错误重置
- 下次启动时，时间戳已相同（元数据已保存），不会触发更新检测
- `isTemplateAvailableLocally()` 依赖过期的 `isLocal` 标志，返回 true
- 结果：使用了旧的缓存文件，没有重新下载

合理的缓存策略应当是：
- 本地缓存用于提升重复访问速度和离线可用性。
- 当检测到远程版本更新时，自动失效本地缓存。
- 始终以文件实际存在性为准，不依赖可能过期的标志。
- 用户无感知地获取到最新版本，无需手动清除缓存。

### How
1. 每次获取远程元数据时，对比 `updatedAt` 时间戳和 MD5 校验值。
2. 如果远程版本比本地缓存新：
   - 删除本地缓存的模板文件
   - 重置本地缓存标记
3. 用户点击模板时，`isTemplateAvailableLocally()` 直接检查文件是否存在：
   - 文件不存在 → 返回 false → 触发下载
   - 文件存在 → 返回 true → 直接使用本地缓存
4. 从缓存加载元数据时，验证文件实际存在才设置 `isLocal=true`
5. 下载完成后，新版本的模板文件被缓存，供后续使用。
