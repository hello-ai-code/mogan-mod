## 5 What
在 `auto-backup-trig-payload` 构造的 payload JSON 里新增 4 个云备份请求头所需的静态字段。

1. `"site"` ← `(current-stem-site)`
2. `"token"` ← `(account-load-token)`
3. `"user-agent"` ← `(stem-user-agent)`
4. `"device-id"` ← `(stem-device-id)`

## 6 Why
autosave 插件是长驻 plugin session，通过 stdin/stdout 管道与主进程通信，无法像 auto-cloud-backup 那样用命令行参数把 token/user-agent/device-id 注入子进程。复用已有的 payload 管道协议传递这些字段，是适配 autosave 进程模型的最自然方案。

数据源函数与 auto-cloud-backup 完全一致（`current-stem-site` / `account-load-token` / `stem-user-agent` / `stem-device-id`），只是传递通道从命令行参数换成 payload JSON。

## 7 How
在 `auto-backup-trig-payload` 的 `let*` 块末尾、`(values ...)` 之前，追加 4 个 `json-push`。每个字段的读取用 `catch #t` 包住对应的主进程访问函数：

- 账号模块未加载、未登录、或 glue 函数未注册时，这些函数会抛异常
- 逐个 catch 回退空串，避免单字段读取失败导致整个 payload 构造中断、进而影响 copy 流程

token 只进入 payload 管道（主进程内存 → 管道 → autosave 子进程内存），不落 manifest、不写日志、不写磁盘，与 auto-cloud-backup "token 只进命令行参数" 的安全语义保持一致。
