## 2026/04/29 保存时触发备份

### What
1. 保存文档时自动触发一次备份，确保用户主动保存的版本也被记录
2. 备份类型标记为 `"save"`，与自动触发的 `"auto"` 和手动触发的 `"manual-open"` 区分
3. 保存触发的备份同样进行 md5 去重检查，避免重复备份相同内容

### 涉及文件
- `TeXmacs/progs/texmacs/texmacs/tm-files.scm`
  - `auto-backup-now` 函数增加 `kind` 参数，支持指定备份类型
  - 保存流程中调用 `auto-backup-now` 时传入 `"save"` 类型
  - `auto-backup-buffer` 支持 `"save"` 类型，跳过 `buffer-modified?` 检查

### Why
- 自动备份的 rolling retention 机制只保留最近 7 个版本
- 如果用户长时间未主动保存，期间可能产生多个自动备份版本
- 保存时触发备份可确保用户的重要保存节点被保留在版本历史中

## 2026/04/28 Auto Backup 落地说明

### 如何测试

1. 设置入口：
   - 打开 Preferences 或启动页 Settings。
   - `Automatically save:` 只应显示 `On` / `Off`，不要向用户展示 `120s`。
   - 社区版按钮文案为 `Open backup folder`；商业版按钮文案为 `Cloud backup`。

2. 新建文档：
   - 新建一个文档
   - 输入内容后点击设置旁按钮，应该生成一次 `manual-open` 备份。
   - 打开 `$TEXMACS_HOME_PATH/system/backup/auto-backup.json`，确认该文档有独立 `doc_id`，版本记录包含 `kind: manual-open` 和 `md5`，文档记录不包含 `user_id`。

3. 打开已有文件：
   - 打开一个没有 `stem-doc-id` 的旧文件，不做编辑、不点击保存。
   - 源文件不应被静默改写；只有用户主动保存时，`stem-doc-id` 才随本次保存写入源文件。
   - 点击设置旁按钮会先做本地备份；如果内容 md5 与最近备份相同，只更新 manifest 检查时间，不新增重复版本。

4. 只读资源：
   - 打开 `TEXMACS_PATH` 目录或子目录下的内置文档。
   - 自动备份和按钮触发都应跳过这类 buffer，`auto-backup.json` 中不应出现对应记录。

5. 30 天清理：
   - 在 `auto-backup.json` 中准备一条 `last_checked_at`、`last_backup_at` 和 `versions[].created_at` 都早于当前时间 30 天以上的记录。
   - 对任意可备份文档触发一次手动备份。
   - 再检查 manifest，旧文档记录应被移除；仍活跃文档中超过 30 天的旧版本也应被移除。

6. 定时触发：
   - 将设置设为 `Off` 后，后续自动调度不再继续安排。
   - 将设置设为 `On` 后，修改文档并等待下一轮自动触发，应在备份目录生成新版本。
   - 未修改的干净文档不会生成周期性备份。

7. 回归检查：
   - 运行 `xmake run tm-files-test`，确认 auto backup 的文件名、doc id、canonical md5、TEXMACS_PATH 跳过、manifest rolling/30 天清理、无 user id 和官网 UTM URL 用例通过。
   - 运行 `git diff --check`，确认没有空白字符问题。


### What
- 保留旧 `~`/`#` autosave 与恢复提示，新增独立 `auto-backup-*` 备份链路。
- 自动保存设置只保留开启（120s）和关闭；开启时旧 autosave 和新自动备份都固定每 120s 调度，关闭时不调度。
- 新自动备份开启后每 120s 扫描 modified buffer，备份到 `$TEXMACS_HOME_PATH/system/backup/`，文件名形如 `abc_yyyyMMddHHmmss.tmu`。
- 新增 `auto-backup.json` manifest，按 `doc_id` 存文档状态，版本列表 rolling 保留 7 个。
- manifest 会在写回前清理 30 天以上的旧文档记录和过期版本，避免 JSON 长期增大。
- 导出的 `.tmu` 只写入 `stem-doc-id`，不写 user id、device id、md5 或客户端版本；manifest 只记录 device id 和客户端版本，md5 仅存在 `versions[].md5`。
- 新建文档会在创建时通过 `init-env` 绑定 `stem-doc-id`；普通保存和自动备份也会兜底确保当前 buffer 有 `stem-doc-id`。已有文件打开时不静默写回源文件，只有用户保存时才持久化。
- 自动保存偏好页和启动设置页保留下拉框，只提供 `On` / `Off`；下拉框右侧只增加一个自动备份入口按钮，按钮文案按版本动态显示。社区版按钮打开本地目录，商业版按钮打开官网并带 `utm_source=auto_backup_button`，点击时都会先对当前 buffer 触发一次 `manual-open` 备份；商业版还会调用预留的自动上传钩子。
- 增加 Scheme 测试 `tm-files-test.scm`，覆盖文件名清理、doc id/canonical md5、manifest retention 与 md5 字段形状。

### How
1. `TeXmacs/progs/texmacs/texmacs/tm-files.scm`
   - 新增 `auto-backup-dir`、`auto-backup-load-manifest`、`auto-backup-save-manifest!`、`auto-backup-buffer`、`auto-backup-upload-buffer`、`open-auto-backup-location` 等 helper。
   - `auto-backup-buffer` 的定时触发先用 `(buffer-modified? name)` 做低成本过滤；按钮触发使用 `"manual-open"`，跳过 modified 过滤但仍做 md5 去重。
   - 保存或备份前通过 `auto-backup-ensure-buffer-doc-id!` 在当前 buffer 绑定缺失的 `stem-doc-id`；打开已有文件时不静默写回源文件，只有用户保存时才持久化到源文件。canonical md5 计算前移除 `stem-doc-id`，避免首次生成 doc id 导致内容误判变化。
   - `auto-backup-buffer-eligible?` 跳过 `TEXMACS_PATH` 目录和子目录下的内置只读文件。
   - manifest 写入使用临时文件再通过 `(liii path)` 的路径操作替换；损坏 manifest 会移动为 `.broken.<timestamp>` 后重建。
   - manifest 每次写回前清理超过 30 天未活动的文档记录，以及仍活跃文档中的过期版本。
   - 调试日志当前打开，方便验证自动备份调度和跳过原因。

2. 时间戳
   - 通过 `(srfi srfi-19)` 的 `current-date` 和 `date->string` 生成 `yyyyMMddHHmmss` 本地时间戳，不新增 timestamp glue。

3. 设置入口
   - `preferences-widgets.scm` 与 `startup-tab-settings.scm` 使用两项下拉框渲染 `On` / `Off`，右侧只渲染一个按钮，按钮 label 由 `auto-backup-button-label` 动态决定：社区版为 `Open backup folder`，商业版为 `Cloud backup`。
   - 启动页 Settings 中点击按钮时，如果 `current-buffer` 是 `tmfs://startup-tab` 等不可备份 buffer，则通过 `auto-backup-manual-target` 选择最近访问的可备份文档，优先选择未保存修改文档。
   - `preferences-menu.scm` 将历史 `Tools -> Autosave` 时长枚举收敛为 `On` / `Off`。

4. 调度入口
   - `init-research.scm` 在启动 idle 后同时安排旧 `autosave-delayed` 和新 `auto-backup-delayed`。

## 2026/04/28 Auto Save 改进设计文档

### 背景
当前自动保存逻辑主要在 `TeXmacs/progs/texmacs/texmacs/tm-files.scm`：
- `autosave-delayed` 根据 `autosave` preference 定时触发，默认值已经是 `120` 秒。
- `autosave-buffer` 把自动保存文件写到源文件旁边，周期保存使用 `~` 后缀，崩溃救援使用 `#` 后缀。
- `load-buffer-check-autosave` 通过比较源文件、`~`、`#` 三者时间戳来提示恢复。
- `save-buffer-save` 成功保存后会调用 `autosave-remove` 删除旧的 `~`/`#` 文件。

新需求要把自动保存从“源文件旁边的临时后缀文件”升级为“统一目录中的版本化备份”，同时为后续自动上传准备元数据。

### 目标
1. 自动保存文件统一写入 `$TEXMACS_HOME_PATH/system/backup/`，文件格式固定为 `.tmu`。
2. 自动保存文件名使用 `<文档名>_<yyyyMMddHHmmss>.tmu`，例如 `abc_20260428115901.tmu`。
3. 自动保存粒度为 120s，每个文档最多保留 7 个 rolling 版本。
4. 在 backup 目录维护一个 JSON 文件，记录当前自动保存状态，并预留后续自动上传所需字段。
5. 文档保存时写入关键元数据：`doc id`、`device id`、`md5` 等。
6. 自动保存设置保留两项下拉框，并在右侧增加自动备份入口按钮：社区版打开 backup 目录，商业版打开官网并附带自动备份按钮来源 UTM。

### 非目标
- 本期不实现自动上传。
- 本期不改普通保存的目标文件路径。
- 本期不把 token、email 等敏感账号信息写入文档或 JSON。
- 本期不改变用户主动保存文档的主流程，只在保存前后补充元数据。

### 用户可见行为
1. 用户开启自动保存后，每 120s 对已修改文档生成一个 `.tmu` 备份。
2. 备份文件统一出现在 `$TEXMACS_HOME_PATH/system/backup/`。
3. 同一个文档最多保留最近 7 份备份，生成第 8 份时删除最旧一份。
4. 如果 TeXmacs 启动或打开文档时发现有比源文件更新的备份，则继续提示用户恢复。
5. 自动保存设置保留 `On` / `Off` 两项下拉框，右侧新增入口按钮；社区版打开 backup 目录，商业版打开官网并触发预留自动上传。

### 存储目录与命名
备份根目录：

```text
$TEXMACS_HOME_PATH/system/backup/
```

索引文件：

```text
$TEXMACS_HOME_PATH/system/backup/autosave.json
```

备份文件：

```text
$TEXMACS_HOME_PATH/system/backup/<safe-base>_<yyyyMMddHHmmss>.tmu
```

命名规则：
- `safe-base` 默认取源文件 basename 去掉后缀，例如 `abc.tm` -> `abc`。
- basename 中不适合文件名的字符替换为 `_`。
- 时间戳使用本地时间，格式固定为 `yyyyMMddHHmmss`。
- 如果同一秒发生重名，优先重试下一秒；仍冲突时追加短序号，JSON 中记录真实路径。

### JSON 结构
使用 `auto-backup.json` 作为 manifest（注意不是 `autosave.json`），采用 object-by-doc-id 结构，方便后续按文档同步。

```json
{
  "meta": {
    "version": 1,
    "interval_seconds": 120,
    "retention": 7,
    "updated_at": 1777358341
  },
  "documents": {
    "<doc_id>": {
      "doc_id": "<doc_id>",
      "display_name": "abc.tm",
      "source_url": "file:///Users/me/doc/abc.tm",
      "format": "texmacs",
      "device_id": "<device-id>",
      "liiistem_version": "v2026.2.3",
      "last_backup_at": 1777358341,
      "last_checked_at": 1777358341,
      "versions": [
        {
          "path": "$TEXMACS_HOME_PATH/system/backup/abc_20260428115901.tmu",
          "created_at": 1777358341,
          "kind": "auto",
          "md5": "<content-md5>",
          "size": 12345,
          "uploaded": false,
          "upload_status": "pending"
        }
      ]
    }
  }
}
```

**注意**：JSON 输出时文档级别不包含 `upload` 字段，只保留版本级别的 `uploaded` 和 `upload_status`。应用版本使用 `XMACS_VERSION` 而非 `TEXMACS_VERSION`。

写入策略：
- 每次自动保存成功后更新 `autosave.json`。
- JSON 写入使用临时文件 + rename，避免崩溃时留下半截 JSON。
- 如果 JSON 损坏，则重建空索引，并可扫描 backup 目录做 best-effort 恢复。
- 版本列表按 `created_at` 从新到旧排序，超过 7 个后删除旧文件并同步 JSON。

### 文档内元数据
只把稳定文档标识放入文档 `initial` collection，避免污染正文。其他自动备份状态只写入 `auto-backup.json`：

```scheme
(associate "stem-doc-id" "<stable-doc-id>")
```

字段说明：
- `stem-doc-id`：文档首次保存或首次自动保存时生成，之后稳定保留。
- 不写入 `stem-content-md5`、`stem-autosave-schema`、`stem-last-saved-at`、user id、device id 或客户端版本到源文档。

**实现细节（重要）**：
- doc-id 通过 `init-env` 写入 buffer 的内存环境变量，而非 `buffer-set` 替换整个文档树。
- 这样可以避免触发文档重新解析和模式检测，解决新建文档时菜单/工具栏变动的问题。
- 保存时 `attach_data` 自动将环境变量持久化到 `initial` 字段；加载时 `detach_data` 自动恢复。
- 读取 doc-id 时优先从 `get-init-env`（内存）获取，如未找到再从文档树 `initial` 字段获取（兼容已保存文档）。

MD5 注意事项：
- 内容 md5 只存储在 `auto-backup.json` 的 `versions[].md5`。
- 计算方式应固定为：对文档 canonical tree 做序列化，并在计算前排除 `stem-doc-id`。
- 自动保存前如果发现新 md5 与最近版本相同，则只更新 autosave 状态，不重复生成备份文件，用来规避当前代码里“仅移动光标也可能触发 autosave”的问题。

### 自动保存流程
1. `autosave-delayed` 保持 120s 粒度触发。
2. `autosave-buffer` 判断 buffer 是否需要自动保存。
3. 确保 backup 目录存在。
4. 确保文档有 `stem-doc-id`，并读取当前 `device_id`。
5. 计算 canonical content md5。
6. 如果 md5 与当前文档最近一个版本相同，调用 `buffer-pretend-autosaved` 并跳过写文件。
7. 生成目标路径，例如 `abc_20260428115901.tmu`。
8. 以 `tmu` 格式导出 buffer 到目标路径。
9. 写入/更新 `autosave.json`。
10. 对该 `doc_id` 执行 rolling retention，只保留 7 个版本。
11. 调用 `buffer-pretend-autosaved`，避免下一轮无变化时重复保存。

崩溃救援场景继续复用 `rescue-mode?`，但不再写 `#` 后缀文件，而是在 JSON 的版本记录里标记：

```json
{ "kind": "rescue" }
```

### 恢复流程
`autosave-propose` 需要改为优先查询 `autosave.json`：
1. 读取当前源文件中的 `stem-doc-id`；如果没有，则用 `source_path_md5` 做兼容匹配。
2. 找到该文档最新备份版本。
3. 如果备份时间晚于源文件最后修改时间，且备份 md5 与源文件当前 md5 不同，则提示恢复。
4. 用户确认后，以 `tmu` 格式导入备份内容，填充到当前 buffer，并标记为 modified。
5. 如果 JSON 不存在或损坏，则回退到旧的 `~`/`#` 检测逻辑，保证旧版本用户仍可恢复。

普通保存成功后不再删除 rolling 备份，只更新 JSON 中的 `last_regular_save_at`。恢复提示以“备份是否比源文件新 + md5 是否不同”为准，避免用户每次打开都被旧备份打扰。
