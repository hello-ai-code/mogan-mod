## 5 What

修复模板下载过程中点击取消后，重新下载同一个模板会导致下载框和文档打开次数"每次递增"的 bug。

1. **`TemplateAPI` 新增 `abortDownload()` 私有方法**
   - 静默终止指定模板的网络请求，清理 `downloadReplies_`
   - **不**发射任何信号，仅用于内部清理

2. **`downloadTemplate()` 改调 `abortDownload()` 清理旧请求**
   - 启动新下载前，用 `abortDownload` 终止同一模板的旧请求
   - 避免旧请求的信号干扰新下载流程

3. **`cancelDownload()` 清理后 emit `downloadFailed`**
   - 用户点击 Cancel 时，终止请求并发射 `downloadFailed(templateId, "Download cancelled")`
   - 确保 `TemplateManager::downloadTemplateSync()` 中的 `QEventLoop` 立即退出

4. **`onDownloadFinished()` 补充 HTTP status code 检查**
   - 原实现仅检查 `reply->error()`，当服务器返回 HTTP 404 但网络传输成功时，`QNetworkReply::error()` 可能仍为 `NoError`
   - 增加对 `HttpStatusCodeAttribute` 的检查，将 `>= 400` 的 HTTP 响应统一视为下载失败

## 6 Why

### 6.1 现象

用户在下载模板时点击取消，再次点击同一模板重新下载，下载完成后：
- 弹出多个（随取消次数递增）下载完成提示
- 同一文档被多次打开

### 6.2 根因

`TemplateAPI::cancelDownload()` 的实现顺序有问题：

```cpp
void TemplateAPI::cancelDownload(const QString& templateId) {
  auto it = downloadReplies_.find(templateId);
  if (it != downloadReplies_.end() && it.value()) {
    disconnect(it.value(), nullptr, this, nullptr);  // 先断开信号
    it.value()->abort();                                // 再 abort
    it.value()->deleteLater();
    downloadReplies_.erase(it);
  }
}
```

- `disconnect` 后调用 `abort()`，`QNetworkReply::finished` 信号发出时，`onDownloadFinished` 已收不到
- `downloadFailed` 信号**永远不会被 emit**
- `TemplateManager::downloadTemplateSync()` 中的 `QEventLoop` 收不到 `downloadCompleted` 或 `downloadFailed`，只能傻等 30 秒超时

在这段超时期间：
- 旧的调用 `downloadTemplateSync()` 的 UI 层对象仍然卡在 `loop.exec()` 中
- 旧对象仍然连接着 `TemplateManager` 的 `downloadCompleted` / `downloadFailed` 信号
- 用户点击"重新下载"后，新的下载完成时会 emit `downloadCompleted`
- **新旧所有卡住的对象都会收到该信号**，导致模板被多次打开

取消 N 次就会累积 N 个"幽灵对象"，下次成功下载时同时触发 N+1 次完成处理。

## 7 How

### 7.1 信号缺失问题

`cancelDownload` 需要 emit `downloadFailed`，让 `downloadTemplateSync` 的 `QEventLoop` 立即退出。但 `downloadTemplate()` 内部在启动新下载前也会调用 `cancelDownload` 清理旧请求，此时 emit `downloadFailed` 会误伤新启动的下载。

### 7.2 拆分为两个方法

| 方法 | 触发场景 | 是否 emit 信号 |
|------|----------|----------------|
| `abortDownload` | `downloadTemplate` 内部启动新下载前 | 否 |
| `cancelDownload` | 用户点击 Cancel 按钮 | 是，emit `downloadFailed` |

```cpp
void TemplateAPI::abortDownload(const QString& templateId) {
  auto it = downloadReplies_.find(templateId);
  if (it != downloadReplies_.end() && it.value()) {
    disconnect(it.value(), nullptr, this, nullptr);
    it.value()->abort();
    it.value()->deleteLater();
    downloadReplies_.erase(it);
  }
}

void TemplateAPI::cancelDownload(const QString& templateId) {
  auto it = downloadReplies_.find(templateId);
  if (it != downloadReplies_.end() && it.value()) {
    disconnect(it.value(), nullptr, this, nullptr);
    it.value()->abort();
    it.value()->deleteLater();
    downloadReplies_.erase(it);
    emit downloadFailed(templateId, tr("Download cancelled"));
  }
}
```

### 7.3 调用链路

**用户取消下载：**
> 注：`QTMTemplateOpener` 是本文档中的概念化表述，对应实际代码中调用
> `TemplateManager::downloadTemplateSync()` 的 UI 层对象（如启动页的模板卡片打开逻辑）。

```
QProgressDialog::canceled
  → UI 层对象::cancelDownload (templateId)
    → TemplateManager::cancelDownload (templateId)
      → TemplateAPI::cancelDownload (templateId)  [emit downloadFailed]
        → TemplateManager::downloadFailed
          → downloadTemplateSync::QEventLoop::quit [立即退出]
```

**重新下载（启动新请求前清理旧请求）：**
```
UI 层对象::openTemplate
  → TemplateManager::downloadTemplateSync
    → TemplateManager::downloadTemplate
      → TemplateAPI::downloadTemplate
        → TemplateAPI::abortDownload (templateId)  [静默清理，不 emit]
```

### 7.4 修复后行为

- 用户点击取消 → `cancelDownload` emit `downloadFailed` → `QEventLoop` 立刻退出 → 调用 `downloadTemplateSync()` 的 UI 层对象立即析构，不残留
- 重新下载同一个模板时，旧的对象已彻底销毁，不会"幽灵"接收新下载的完成信号
- `downloadTemplate` 内部调用 `abortDownload` 清理旧请求，不会误 emit `downloadFailed` 导致新下载被中断
