# 216_23 metadata.json ETag 条件请求缓存

## 如何测试

1. 编译：`xmake b stem`
2. 打开启动页 `Template` 页面，等待模板列表加载完成。
3. 找到缓存目录（通常为 `~/.local/share/moganlab/system/template_cache/`），确认存在 `metadata_etag.txt` 文件，内容为一个带引号的字符串（如 `"69e6dbe2-2ee1"`）。
4. 关闭软件后重新打开（触发 `refreshTemplates`）：
   - 观察控制台输出 `"Metadata not modified (304), using cached data"`，表示服务器返回 304，未重新下载 metadata.json。
   - 模板列表应正常显示，数据来自本地缓存。
5. 测试 ETag 失效（本地模拟文件更新）：
   - 修改 `metadata_etag.txt` 的内容为一个错误值（如 `"wrong-etag"`）。
   - 关闭启动页后重新打开 `Template` 页面。
   - 验证：服务器返回 200，重新下载 metadata.json，且 `metadata_etag.txt` 被更新为新的 ETag。
6. 测试离线/网络异常场景：
   - 断开网络后打开 `Template` 页面。
   - 验证：metadata.json 请求失败，但 UI 仍显示上次缓存的模板列表。
7. 测试 ETag/缓存不一致的边界情况：
   - 手动删除 `metadata.json` 但保留 `metadata_etag.txt`。
   - 重新打开 `Template` 页面。
   - 验证：自动清除 ETag 并重新发起无条件的完整请求，UI 正常显示模板列表（而非空列表或卡住）。

## 2026/04/21 实现说明

### What

本次为 metadata.json 添加了 HTTP 条件请求（ETag / `If-None-Match`）支持，构成模板中心三层缓存体系的最后一环：

| 层级 | 缓存机制 | 状态 |
|------|---------|------|
| metadata.json | HTTP ETag（本次新增） | 避免重复下载/解析整个列表 |
| .tmu 模板文件 | `updated_at` + `file_md5` 比对（已有） | metadata 层已决策是否更新 |
| 预览 PDF | HTTP ETag + `If-Modified-Since`（216_22 已实现） | 避免重复下载 PDF |

#### 修改文件

**src/Mogan/TemplateCenter/template_cache.hpp** (修改)
- 新增 `metadataEtag()` / `setMetadataEtag()` 公有方法，用于读写 metadata.json 的 ETag。
- 新增 `metadataEtagPath()` 私有方法，返回 `metadata_etag.txt` 路径。
- 新增 `metadataEtag_` 内存缓存成员（声明为 `mutable`，使 `metadataEtag()` const 方法可直接写入，无需 `const_cast`）。
- 从 `CacheEntry` 结构体中移除未使用的 `etag` 字段（`.tmu` 的更新检测由 metadata 层的 `updated_at` + `file_md5` 负责，无需额外的 HTTP ETag）。

**src/Mogan/TemplateCenter/template_cache.cpp** (修改)
- 实现 ETag 持久化：读写 `metadata_etag.txt` 文件。
- `clearCache()` 同步删除 `metadata_etag.txt` 并清空内存缓存，确保清除缓存后下次走完整下载流程。
- 从 `loadCacheIndex()` 和 `saveCacheIndex()` 中移除 `etag` 的读写逻辑，减少无效代码。
- `setMetadataEtag()` 在 `QFile::open()` 失败时输出 `qWarning`，避免静默丢失 ETag。

**src/Mogan/TemplateCenter/template_api.hpp** (修改)
- 新增 `setMetadataEtag(const QString& etag)` / `lastMetadataEtag()` 方法。
- 新增 `metadataNotModified()` 信号，用于通知上层 304 响应。
- 新增 `metadataEtag_`（发送用）和 `lastMetadataEtag_`（接收用）成员变量。

**src/Mogan/TemplateCenter/template_api.cpp** (修改)
- `fetchMetadata()`：若存在缓存 ETag，在请求头中附加 `If-None-Match`。
- `onMetadataReplyFinished()`：
  - **先检查 HTTP status code**，若 304 则发射 `metadataNotModified()` 并返回；**此检查在 `error()` 之前**，避免某些 Qt 版本将 304 报告为网络错误而导致无法进入 304 分支。
  - 200 响应时从 `ETag` 头提取新值，保存到 `lastMetadataEtag_` 供上层持久化。

**src/Mogan/TemplateCenter/template_manager.hpp** (修改)
- 新增 `onMetadataNotModified()` slot，处理 304 场景。

**src/Mogan/TemplateCenter/template_manager.cpp** (修改)
- 构造函数中连接 `TemplateAPI::metadataNotModified` 信号。
- `refreshTemplates()`：请求前从 `TemplateCache` 读取 ETag 并注入 `TemplateAPI`。
- `onRemoteMetadataLoaded()`：收到 200 后，将 `lastMetadataEtag()` 保存到 `TemplateCache`（无论是否为空，确保服务端撤销 ETag 时本地同步清空）。
- `onMetadataNotModified()`：收到 304 后，若 `templates_.isEmpty()`（ETag 文件存在但缓存数据缺失），当场清除 ETag 并调用 `refreshTemplates()` 重新发起无条件的完整请求；否则直接通知 UI `templatesLoaded()` 复用已有缓存数据。

### Why

1. **减少无效请求**：metadata.json 每次打开模板中心都会无条件下载，即使内容未变更。服务端（liiistem.cn / Nginx）已支持 ETag，完全可以通过 304 避免重复传输。
2. **与现有体系对齐**：PDF 预览层（216_22）已实现完整的 HTTP 条件请求，metadata.json 作为更高频的请求入口，同样值得优化。
3. **前置决策 vs 双重确认**：`.tmu` 文件的更新检测依赖 metadata 中的 `updated_at` 和 `file_md5`，在 metadata 层已经决策完毕，因此 `.tmu` 不需要额外的 ETag。而 PDF 预览没有独立的元数据变更信息，必须依赖 HTTP 条件请求。metadata.json 介于两者之间——它是自动高频请求，且服务端能提供 ETag，所以加一层 ETag 缓存收益最高。

### How

1. **ETag 持久化**：`TemplateCache` 将 ETag 保存为独立的 `metadata_etag.txt` 文件，与 `metadata.json` 同目录，读写开销极小。
2. **网络层条件请求**：`TemplateAPI` 在发起 `GET /templates.json` 时，若本地有 ETag 则附加 `If-None-Match` 头；收到 200 时提取响应头中的新 ETag。
3. **协调层处理**：`TemplateManager` 作为中间层，负责"请求前注入 ETag → 收到 200 保存新 ETag → 收到 304 直接复用缓存"的完整生命周期。
4. **边界情况**：
   - 服务端未返回 ETag：清空本地 ETag 缓存（`metadata_etag.txt`），下次请求不再发送 `If-None-Match`，避免客户端状态与服务端能力脱节。
   - 首次启动无缓存：不发 `If-None-Match`，走完整下载流程。
   - 清除缓存：`clearCache()` 同时删除 `metadata_etag.txt`，下次走完整下载。
   - ETag/缓存不一致（如手动删除 `metadata.json` 但保留 `metadata_etag.txt`）：收到 304 但无缓存数据时，自动清除 ETag 并重新发起完整请求。
   - 无网络且无缓存：UI 显示 "No templates available"（暂无可用模板）。
