### 3.2 非确定性测试（文档验证）

8. 点击主编辑器区域右侧按钮打开的侧边栏 chat，使用的是和 chat tab 一致的session

## 5 What

延迟注册机制：点击"新会话"时只创建内存中的 session + UI 面板，不注册到持久化层、不加入侧边栏。用户首次发送消息时才完成注册。

1. `ChatSession` 新增 `registered` 字段，标记是否已注册到持久化层
2. 新增 `registerSession()` 方法，在首次发送时完成注册（Scheme 持久化 + 侧边栏 item + buffer-pretend-saved）
3. `ensureNewConversation()` 不再调 `chat-persist-register-session`，通过 `findReusableSession` 复用空白会话
4. `exportBuffer()` / `updateManifest()` 对未注册 session early return
5. `onThinkingToggled()` 对未注册 session 只更新内存
6. `onDeleteRequested()` 对未注册 session 跳过 `chat-persist-delete-one`
7. `restoreSessionMeta()` 标记从磁盘恢复的 session 为 `registered=true`
8. 补充 `registered` 字段的单元测试

## 6 Why

用户点击"新会话"后立即注册到 Scheme 持久化层并加入侧边栏，导致：
1. 用户只是点了一下就切走，侧边栏出现空"新会话" item
2. 磁盘上留下空 session 文件（message.tmu 为空）
3. 重启后看到一堆空白会话

## 7 How

### 7.1 核心机制：registered 字段

```
                    点击"新会话"
                         │
                    createSession()
                    registered = false
                         │
               ┌──── 用户输入并发送 ────┐
               │                        │
         registerSession()        切到已有会话 /
         registered = true        再次点"新会话"
         加入 sidebar              │
         注册持久化           findReusableSession()
                              复用该空白会话
                              输入内容保留
```

### 7.2 方法职责划分

| 方法 | registered=false 时行为 | registered=true 时行为 |
|------|------------------------|----------------------|
| `registerSession` | 注册 Scheme + 加入 sidebar + 标记 true | no-op |
| `exportBuffer` | early return | 正常导出 buffer |
| `updateManifest` | early return | 正常更新 manifest |
| `onThinkingToggled` | 只更新内存 `thinking` 字段 | 更新内存 + Scheme + manifest |
| `onDeleteRequested` | 跳过 `chat-persist-delete-one` | 完整删除流程 |
| `onArchiveRequested` | `is_empty(title)` 跳过 | 正常归档 |

### 7.3 复用而非删除

切换到已有会话时，不删除空白会话面板（`deleteLater` 会导致排版器 SIGSEGV），而是利用 `QStackedWidget` 的特性：激活其他面板时，空白会话面板自动隐藏。再次点击"新会话"时，`findReusableSession` 找到该空白会话并复用，输入内容自然保留在 buffer 中。

### 7.4 不涉及面板删除

`discardUnregisteredBlankSession` / `detachPanel` 等方案均尝试删除/分离面板，但 `deleteLater` 导致排版器访问已释放内存崩溃。最终采用"不删除、只复用"的安全策略：空白会话面板始终存活，直到用户发送消息（注册）或应用关闭。
