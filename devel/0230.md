## 5. 2026/05/27 统一流式 HTTP 返回值为 hashtable

### 5.1 What
1. goldfish.hpp: 新增 `error2hashtable` 辅助函数（异常时构造错误 hashtable）
2. goldfish.hpp: `f_http_post`/`f_http_get` 流式模式始终返回 `response2hashtable`，移除 `s7_undefined` 和 `s7_make_integer`
3. goldfish.hpp: catch 分支统一用 `error2hashtable(sc, 0, url, e.what())` 替代 `s7_make_integer(sc, 500)`
4. llm-chat.scm: `response2either` 移除 `(undefined? r)` 分支，`stream-completed?` 检查移入 `(= status-code 200)` 分支
5. http-post-test.scm: 更新 4 个测试（流式 200 应返回 hashtable 而非 undefined），新增 3 个状态码测试

### 5.2 Why
原来流式模式下 `g_http-post` 始终返回 `s7_undefined`，丢弃 response 对象：
- 非 200 错误（401/403/500）拿不到真实状态码 → 误报 "HTTP状态码为0"
- 异常时返回 `s7_make_integer(500)` 类型不一致 → Scheme 层 `(r 'status-code)` 会崩溃

### 5.3 How
统一所有返回路径为 hashtable：
- 正常：`response2hashtable(sc, response)`
- 异常：`error2hashtable(sc, 0, url, e.what())`

Scheme 层 `response2either` 不再需要 `(undefined? r)` 分支，所有情况都通过 `(r 'status-code)` 判断：
- `200` + `stream-completed?=true` → 成功
- `200` + `stream-completed?=false` → 断网（流式中途中断）
- `0` → 连接失败（DNS/网络不可达）
- `401/403` → 对应业务错误
- 其他 → 通用 HTTP 错误
