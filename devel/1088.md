# [1088] 修复 Windows 上导出带图片 HTML 时 base64 为空

## 如何测试
在 Windows 上打开包含嵌入图片的 .tm 文件，导出为 HTML（带 base64 内联图片），检查导出的 HTML 中图片是否正常显示。

## 2026/06/12

### What
修复 Windows 上导出包含嵌入图片（PDF 格式）的 HTML 时，图片 base64 数据为空或损坏的问题。

### Why
在 macOS 上导出正常，但在 Windows 上导出的 HTML 图片显示为空白或 unknown 占位图。

经排查发现三个独立但相互关联的问题：

1. **`tmhtml-read-binary-file` 使用 `open-input-file` 文本模式读取 PNG**：Windows 文本模式遇到二进制数据中的 `0x1A` (EOF 标记) 会提前终止读取，导致 1.3MB 的 PNG 只读到 5 字节。macOS/Linux 无此问题。

2. **`tmhtml-write-binary-file` 使用 `open-output-file` 文本模式写入 PDF**：`\n` 被转换为 `\r\n`，导致 PDF 文件损坏，MuPDF 无法正确渲染。macOS/Linux 无此问题。

3. **嵌入 PDF 图片路径使用 `url->unix` 格式**：在 Windows 上产出 `/C/Users/...` 格式的路径，排版系统内部 `concretize` 无法正确解析，导致 `mupdf_load_image` 找不到文件，fallback 到 `unknown.png`。

### How
修改三个文件：

1. **`TeXmacs/plugins/html/progs/convert/html/tmhtml.scm`**：
   - `tmhtml-read-binary-file`：改用 `(liii path)` 的 `path-read-bytes`，底层 C++ 用 `tb_file_init` 以二进制模式读取
   - `tmhtml-write-binary-file`：改用 `(liii path)` 的 `path-write-bytes`，底层 C++ 用 `tb_file_writ` 直接写入二进制数据
   - 嵌入 PDF→PNG 路径：将 `(url->unix abs-url)` 改为直接使用 `(url-concretize name-url)` 的结果，确保在所有平台上使用原生路径格式

2. **`TeXmacs/plugins/goldfish/src/goldfish.hpp`**：
   - 新增 `f_path_write_bytes` / `g_path-write-bytes` C++ 函数，接收 Scheme bytevector 并以二进制模式写入文件

3. **`TeXmacs/plugins/goldfish/goldfish/liii/path.scm`**：
   - 新增 `path-write-bytes` 导出，包装 `g_path-write-bytes`
