# 216_33 Recent 列表 tooltip 闪烁修复与 Windows 重复记录去重

## 如何测试

1. 编译：`xmake b stem`
2. 启动 Mogan，打开启动页 **File** 页面。
3. **Tooltip 稳定性**：将鼠标悬停在任意 Recent 条目上，保持 5 秒以上，确认 tooltip **不闪烁、不抖动**，完整路径稳定显示。
4. **Windows 重复记录**（Windows 环境）：
   - 打开启动页 **Template** 页面，点击任意本地模板并使用。
   - 返回启动页 **File** 页面，确认该模板仅出现 **1 条** Recent 记录。
   - 检查 `recent-files.json`，确认同一文件没有同时存在 `C:\` 和 `C:/` 两种路径格式。
5. **File 菜单打开**：通过 `File -> Open a Document` 打开一个文件，确认启动页 Recent 中该文件仅出现 1 条。
6. **旧数据兼容**：若本地 `recent-files.json` 中已存在混合格式的重复路径（如既有 `\` 又有 `/`），重启后加载应自动合并为单条。
7. **实时刷新仍可用**：切到 Template 或 Settings 页面后再切回 File，Recent 列表应立即反映最新状态。
8. **Template 页面列数**：启动 Mogan 后首次进入 **Template** 页面，确认模板卡片列数直接显示正确（不出现先显示 4 列再闪回 6 列的情况）。调整窗口宽度后列数应正确自适应。

## 2026/04/24 实现说明

### What

修复启动页 File 页面 Recent 板块的两个问题，并修复 Template 页面初次进入时列数不正确的问题：
1. **Tooltip 闪烁**：鼠标悬停时 tooltip 以约 1 秒间隔频繁消失重现，视觉体验极差。
2. **Windows 重复记录**：在 Windows 下通过 Template 页面打开文档时，Recent 中会出现两条相同文件（路径格式不同：一条为 `C:/Users/...`，一条为 `C:\Users\...`）。
3. **Template 页面初次列数错误**：启动后首次进入 Template 页面时，列数会先显示默认值（4 列），闪一下后才变为正确列数（如 6 列）。

#### 修改文件

**src/Plugins/Qt/qt_file_page.cpp**
- **移除 `recentRefreshTimer_`（1 秒轮询定时器）** 及相关 `showEvent`/`hideEvent` 启停逻辑。
- `loadRecentDocs()`：
  - 从 Scheme 获取路径后，统一调用 `QDir::fromNativeSeparators` 将反斜杠标准化为正斜杠。
  - 使用 `QStringList::removeDuplicates()` 对路径列表进行去重（保留首次出现的顺序）。
- `addRecentDoc()` / `removeRecentDoc()`：对入参路径统一标准化后再匹配和操作，确保内存列表、Scheme 状态、JSON 保存三者路径格式一致。
- `saveRecentDocs()`：读取 JSON 现有记录时也对 `path` 字段进行标准化，避免旧数据中的混合格式导致重复键。
- 移除未使用的常量 `kRecentRefreshMs`。

**src/Plugins/Qt/qt_file_page.hpp**
- 移除 `QTimer* recentRefreshTimer_` 成员及 `QTimer` 前向声明。

**src/Plugins/Qt/qt_startup_tab_widget.cpp**
- 模板打开连接的 lambda 中，**移除 `filePage_->addRecentDoc(filePath)`** 调用，仅保留 `eval_scheme("(load-document ...)")`。

**src/Plugins/Qt/qt_template_page.cpp**
- `showEvent()`：不再单纯依赖 `gridNeedsRefresh_` 标志，而是在页面真正可见时重新调用 `calculateColumnCount()`。
  - 若 `gridNeedsRefresh_` 为 true，或者计算出的新列数与当前 `currentColumnCount_` 不一致，则重新 `refreshTemplateGrid`。

### Why

#### 1. 为什么移除 216_16 引入的 1 秒轮询刷新

216_16 为了解决“停留在 File 页面时，通过其他入口打开文档后 Recent 不更新”的问题，引入了 `showEvent` 启动、`hideEvent` 停止的 **1 秒 `QTimer`**，周期性调用 `refreshRecentDocs()` → `loadRecentDocs()` → `renderRecentDocs()`。

该方案带来了严重的副作用：
- `renderRecentDocs()` 的实现是**完全清空并重建**整个 `QListWidget`（`takeItem` + `delete widget` + 重新 `addItem` + `setItemWidget`）。
- 当鼠标悬停在某个条目上、tooltip 正在显示时，若恰好触发 timer，`QListWidgetItem` 被销毁重建，tooltip 会强制消失；timer 再次触发前用户未移开鼠标，tooltip 又会重新显示。这导致 **tooltip 以约 1 秒周期闪烁**。
- 此外，每秒重建 50 个带自定义 widget 的列表项是完全不必要的性能开销。

**替代方案**：移除定时器后，Recent 的刷新时机为：
- `showEvent`：页面显示时立即刷新（覆盖切页场景）。
- `addRecentDoc` / `removeRecentDoc`：操作后立即刷新（覆盖当前页内打开/删除文档的场景）。
- 通过其他入口（如 File 菜单）打开文档时，由于用户焦点已离开启动页，回到启动页时会触发 `showEvent` 刷新。

这一机制已能覆盖 216_16 想要解决的全部场景，且避免了闪烁和性能浪费。

#### 2. 为什么 Windows 会出现重复记录

在 `qt_startup_tab_widget.cpp` 中，模板打开时的原逻辑为：
```cpp
filePage_->addRecentDoc(filePath);   // ① Qt 层以 C:/Users/... 格式添加
eval_scheme("(load-document ...")     // ② Scheme 层内部调用 buffer-notify-recent
```

而 `load-document` 的调用链最终会执行 `load-buffer-open`，其中固定调用：
```scheme
(buffer-notify-recent name)
```
该函数使用 `(url->system name)` 生成路径字符串。在 Windows 上，`url->system` 的输出使用**反斜杠**（`C:\Users\...`）。

于是 Qt 侧和 Scheme 侧分别添加了一次同文件，但路径格式不同。由于 `addRecentDoc` 的匹配逻辑是精确字符串比较，两者被视作不同条目，导致列表中出现重复。

#### 3. 为什么只移除 Qt 侧的 `addRecentDoc`，而不是 Scheme 侧的 `buffer-notify-recent`

`buffer-notify-recent` 是 TeXmacs 核心文件操作的一部分，被 `load-buffer`、`save-buffer` 等多个路径共用，改动影响面极大。而 `filePage_->addRecentDoc` 仅存在于启动页的模板打开连接中，是局部冗余。移除后：
- Scheme 侧的 recent list 由 `load-document` 正常维护。
- Qt 侧的 `showEvent` 会从 Scheme 同步最新状态，两者保持一致。

#### 4. 为什么统一标准化为 `QDir::fromNativeSeparators`

即使移除了双重添加，本地 `recent-files.json` 中可能仍残留旧数据（之前版本写入的 `C:\` 格式路径）。统一标准化后：
- 新旧数据在加载时都会被转为 `C:/` 格式。
- 内存列表的键匹配不再受分隔符差异影响。
- 保存回 JSON 时也只写入正斜杠格式，逐步消除历史混合格式。

### How

- **去定时器**：删除 `recentRefreshTimer_` 及相关代码，将刷新收敛到 `showEvent` 和显式操作后调用。
- **去冗余添加**：`qt_startup_tab_widget.cpp` 模板打开时不再手动 `addRecentDoc`，完全委托 `load-document` 的 Scheme 侧逻辑。
- **路径标准化**：在 Qt 侧所有涉及路径比较、保存、Scheme 通信的位置，统一使用 `QDir::fromNativeSeparators`。
- **加载时去重**：`loadRecentDocs` 先统一路径格式，再调用 `QStringList::removeDuplicates()` 进行去重，作为兜底保护。
