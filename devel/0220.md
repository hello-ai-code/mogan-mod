## 5 What

1. Cancel 策略从 `connection-interrupt` + `plugin-cancel` 改为 `connection-stop`，彻底终止 LLM 子进程
2. 发送消息时通知 C++ 进入 Generating 状态（`chat-tab-notify-state "generating"`）
3. Cancel 后手动通知 C++ 恢复 Idle 状态（kill 子进程后 plugin 完成回调不再触发）
4. C++ `notifyStateChanged` 中增加 `btn->setIcon()` 切换按钮图标（send.svg / cancel.svg）
5. `onSendRequested` 增加 Generating 状态检查，防止生成中重复发送
6. `onCancelRequested` 移除冗余的 `setState(Idle)`，状态由 Scheme 侧统一管理
7. 兜底 default model 从 `"default"` 改为 `"Kimi-VLM"`
8. 修复 `images.qrc` 合并冲突，保留双方新增的 SVG 资源

## 6 Why

原有 cancel 使用 `connection-interrupt` 仅设置中断标志，LLM 子进程内部的 HTTP 流式请求继续运行，无法真正终止生成过程。改进方案用 `connection-stop` kill 子进程，利用插件系统已有的自动重启能力（`plugin-do` 检测到 DEAD 状态后自动调用 `plugin-start`）。

此外，C++ 侧的 `notifyStateChanged` 只切换了 tooltip 和信号连接，遗漏了 `setIcon` 调用，导致按钮图标在状态变化时不切换。Scheme 侧发送消息后也从未通知 C++ 进入 Generating 状态。

## 7 How

### 状态同步流程

```
发送消息:
  C++ onSendRequested
  → Scheme chat-tab-send → chat-tab-session-feed
    → chat-tab-notify-state "generating"  ← 新增
    → C++ notifyStateChanged
      → 切换图标 cancel.svg + tooltip Stop + 信号连到 onCancelRequested

取消:
  C++ onCancelRequested
  → Scheme chat-tab-cancel
    → connection-stop (kill 子进程)
    → chat-tab-notify-state "idle"  ← 新增
    → C++ notifyStateChanged
      → 切换图标 send.svg + tooltip Send + 信号连到 onSendRequested

正常完成:
  plugin 完成回调 → qt-chat-tab-set-state "idle"
  → C++ notifyStateChanged → 切换回 send 图标
```

### 关键改动

1. **chat-tab-cancel**: `connection-stop` 替代 `connection-interrupt` + `plugin-cancel`，kill 后手动通知 "idle"
2. **chat-tab-session-feed**: 在 `plugin-feed` 之前调用 `chat-tab-notify-state "generating"`，确保按钮及时切换
3. **notifyStateChanged**: 新增 `setIcon` 调用，Generating 时用 `cancel.svg`，Idle 时用 `send.svg`
4. **onSendRequested**: 开头检查 Generating 状态直接 return，防止生成中触发发送
