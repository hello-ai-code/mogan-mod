# 200_64 Telemetry（埋点）系统

## 如何测试

### 1. 编译与启动
```bash
xmake config --is_community=false
xmake b stem
xmake r stem
```

### 2. Telemetry 开关
```scheme
(set-preference "telemetry" "0")  ; 关闭
(set-preference "telemetry" "1")  ; 开启（默认）
```

### 3. 数据落盘
当前埋点的可测试项：
- 智能粘贴（ctrl/command + shift + v）
- OCR识别（图片悬停按钮 或 智能粘贴图片）

触发 300 次以上埋点事件（或等待 5 分钟），查看终端输出：
```
[telemetry] flush: N events -> detail-telemetry-20250509-143052-1.jsonl
```

检查 `$TEXMACS_HOME_PATH/system/telemetry/main/` 目录，每次 flush 生成一个独立 jsonl 文件；`main-telemetry.json` 记录最近 200 个文件名。

### 4. Scheme 直接调整参数与手动触发 flush
可以直接查看当前设置的值：
```scheme
;; 获取 buffer-size
(telemetry-get-buffer-size)
;; 获取 flush 间隔（单位 ms）
(telemetry-get-flush-interval)
```

在软件内通过 Scheme 直接修改 buffer-size 和 flush-interval（无需重启）：
```scheme
;; 缩小 buffer-size 以便快速触发自动 flush
(telemetry-set-buffer-size! 2)
;; 缩短 flush 间隔（单位 ms）
(telemetry-set-flush-interval! 5000)
```
验证：触发 2 次 `track-event` 后队列自动清空（终端输出 flush 日志）。
恢复默认：
```scheme
(telemetry-set-buffer-size! 300)
(telemetry-set-flush-interval! 300000)
```

**手动强制 flush（调试用）：**
```scheme
;; 强制将当前内存队列写入磁盘（不依赖 buffer-size 和定时器）
(telemetry-flush)
```

**快速验证独立文件写入（一次性测试）：**
```scheme
;; 1. 确保 telemetry 开启
(set-preference "telemetry" "1")
;; 2. 清空队列
(set! *telemetry-event-queue* '())
;; 3. 手动 track 2 个事件
(track-event "TEST_EVENT_A" '())
(track-event "TEST_EVENT_B" '(("foo" . "bar")))
;; 4. 强制 flush
(telemetry-flush)
;; 5. 查看生成的文件
(telemetry-read-meta)        ;; 读取 meta，返回最近 200 条文件名列表
(telemetry-full-path "...")  ;; 拼接完整路径
```

### 5. 社区版兼容性
```bash
xmake config --is_community=true
xmake b stem
```
确认编译通过，无埋点日志输出，`telemetry_track` 为空操作，`telemetry-enabled?` 返回 `#f`。

### 6. 自动化测试
```bash
./bin/test_only 200_64
```

## 2026/05/09 改为独立 jsonl 文件 + 原子 meta 更新

### 动机

原方案使用单文件 `telemetry-pending.jsonl` 追加写 + 目录锁机制，锁逻辑复杂且跨平台容易出 bug。改为每次 flush 生成独立 jsonl 文件，彻底消除文件级竞争。

### 目录结构调整

```
$TEXMACS_HOME_PATH/system/telemetry/
└── main/
    ├── main-telemetry.json                    ← 原子写入，滚动 200 条
    ├── detail-telemetry-20250509-143052-1.jsonl
    └── ...
```

### 核心变更

| 项 | 旧方案 | 新方案 |
|---|---|---|
| flush 间隔 | 60 秒 | **5 分钟（300000ms）** |
| 写入方式 | 单文件追加写 | **每次 flush 独立 jsonl** |
| 文件名 | `telemetry-pending.jsonl` | `detail-telemetry-YYYYMMDD-HHMMSS-序号.jsonl` |
| 竞争控制 | 目录锁（`mkdir` + owner.json） | **无锁，天然隔离** |
| meta 文件 | 无 | `main-telemetry.json`，滚动 200 条 |
| 原子性 | 锁保证 | **`path-unlink` + `path-rename` 原子更新 meta** |
| 空队列 flush | 写空文件 | **跳过，不生成文件** |
| 旧文件清理 | 由 liii 子进程处理 | **mogan meta 滚动时删除（超 200 条删最旧）** |

### 消费者约定（goldfish）

- 只读 `main-telemetry.json`，不写入
- 与 `cloud-telemetry.json` diff 获取待发送文件列表
- 启动时 cloud文件 与 main文件 对齐，发送后追加

## 2026/05/07 初步设计

### 新增文件

| 文件 | 职责 |
|---|---|
| `src/Mogan/Telemetry/telemetry.hpp` | C++ 胶水层。非社区版调用 Scheme `track-event`；社区版为空操作。 |
| `TeXmacs/progs/telemetry/telemetry-utils.scm` | 路径、配置、设备信息、事件构造。 |
| `TeXmacs/progs/telemetry/telemetry-track.scm` | 事件入队、内存队列、flush（锁 + 追加写 JSON Lines）。复杂清理逻辑（大小限制、过期过滤）由 liii 子进程负责。 |
| `TeXmacs/progs/telemetry/init-telemetry.scm` | 启动时加载配置、注册周期性 flush 与退出 flush 钩子。 |
| `TeXmacs/tests/200_64.scm` | 自动化测试：事件构造、开关、配置 clamp。 |

### 修改文件

| 文件 | 修改 |
|---|---|
| `xmake.lua` | include 路径新增 `src/Mogan/Telemetry`。 |
| `TeXmacs/progs/init-research.scm` | 启动流程末尾引入并初始化 telemetry 模块。 |

### 核心机制

- **事件格式**：`telemetry-make-event` 统一生成包含 `eventType`、`timestamp`、`distinctId`、`sessionId`、`eventId`、`appVersion`、`deviceId`、`platform`、`language`、`timezone`、`properties` 的标准结构。
- **内存缓冲**：`track-event` 将事件写入内存队列，达到 `telemetry-buffer-size`（默认 300）后自动 flush。
- **周期性刷新**：每 60 秒 delayed 任务检查并 flush。
- **退出 flush**：`on-exit` 钩子确保关闭前落盘。
- **多进程安全**：目录锁（`mkdir` 原子性）防止并发写入损坏。
- **主进程轻量**：`telemetry-write-pending` 只做追加写（`string-append-to-file`），不读取旧文件、不解析 JSON、不过滤、不检查大小。复杂清理由 liii 子进程异步处理。
- **兜底措施**：
  - 内存队列上限 1000，超出丢弃最旧事件
  - 写入失败保留内存队列，下次重试
  - `on-exit` flush 包裹 `catch`，防止异常中断退出链
  - 锁 owner 不匹配时输出 warn 日志便于排查
- **社区版隔离**：`telemetry-enabled?` 在 Scheme 层检测社区版并返回 `#f`；C++ 层 `#if !IS_COMMUNITY` 编译期剔除 `telemetry_track`。
