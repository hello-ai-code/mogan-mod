## 5 What
将 `goldfish` 仓库提交 `681800d6ebd0d68530ed53c7d7913fd5d7a9a97e` 中 `path-copy` 相关实现完整同步到 `mogan` 内置的 Goldfish。

1. 在 `(liii path)` 中新增 `path-clone`，保留原来复制 path 对象的语义
2. 将 `path-copy` 改为文件复制语义，并新增 `path-copy-into`
3. 在 C 层新增 `g_path-copy`，供 Scheme 层的文件复制接口调用
4. 新增 `path-clone`、`path-copy`、`path-copy-into` 三个测试文件
5. 实际编译并运行三个 Scheme 测试，确认迁移后的实现可用

## 6 Why
`mogan` 中的 Goldfish 仍保留旧语义：`path-copy` 复制的是 path 对象，而不是文件内容。这与上游 `goldfish` 的新接口定义不一致，也会让依赖新 API 的代码无法直接同步。

因此需要将上游已经完成的 `path-copy`/`path-copy-into` 设计同步进来，同时保留原对象复制能力为 `path-clone`，以对齐接口语义和测试覆盖。

## 7 How
完全参考上游提交实现，不额外改写设计：

1. Scheme 层：
   - `path` 构造函数在接收 path 参数时，调用 `path-clone`
   - 原 `path-copy` 重命名为 `path-clone`
   - 新 `path-copy` 调用 `g_path-copy` 执行文件复制
   - 新 `path-copy-into` 通过 `path-name` 和 `path-join` 生成目标路径

2. C 层：
   - 在 `goldfish.hpp` 中新增 `f_path_copy`
   - 使用 `tb_file_init`、`tb_file_read`、`tb_file_writ` 实现源文件到目标文件复制
   - 在 `glue_liii_path` 中注册 `g_path-copy`

3. 测试：
   - `path-clone-test.scm` 验证 path 值复制语义
   - `path-copy-test.scm` 验证文件复制到指定路径
   - `path-copy-into-test.scm` 验证复制到目录并保留原文件名
