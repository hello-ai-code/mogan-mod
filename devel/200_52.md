# [200_52] 启动时忽略用户设置的 TEXMACS_PATH，始终使用软件自带路径

## 如何测试

此 PR 仅在 Windows、macOS 平台下测试

设置错误的 `TEXMACS_PATH` 时，可按自己使用的 shell 选择对应命令：

- elvish: `set-env TEXMACS_PATH /tmp/not-exist`
- PowerShell: `$env:TEXMACS_PATH = "/tmp/not-exist"`

测试项一：Windows 启动时覆盖用户环境中的 TEXMACS_PATH
1. 先在系统或启动终端中设置一个错误的 `TEXMACS_PATH`，或者是直接安装TeXmacs 2.1.5
2. 启动 Mogan
3. 软件应能够正常启动，并使用可执行文件所在目录推导出的程序内置路径，而不是用户环境中的值

测试项二：内置插件目录应跟随程序最终生效的 TEXMACS_PATH
1. 在当前 shell 中先设置一个错误的环境变量，例如 `/tmp/not-exist`
2. 启动 Mogan
3. 进入 Scheme 会话，执行 `(find-binary-goldfish)`
4. 返回结果应指向程序自带目录下的 `plugins/goldfish/bin/goldfish` 或 `plugins/goldfish/bin/goldfish.exe`，而不是用户环境变量对应的错误目录

测试项三：插件不应受本次修改影响
1. 启动 Mogan，进入 Scheme 会话，执行 `(getenv "TEXMACS_HOME_PATH")`，记下程序显示的实际用户目录路径
3. 确认该目录下存在 `plugins/python/bin/python.pex`
4. 在当前 shell 中先设置一个错误的环境变量，例如 `/tmp/not-exist`
5. 启动 Mogan
6. 进入 Scheme 会话，执行 `(python-launcher)`
7. 返回的启动命令应指向上一步目录下的 `plugins/python/bin/python.pex`，证明本次修改并未影响插件系统

测试项四：macOS 开发环境不应受本次修改影响
1. 拉取本次修改的最新代码
2. xmake b stem ; xmake r stem
3. 软件应正常启动

## 2026/04/02 修复 macOS 平台开发环境的 TEXMACS_PATH

## 2026/04/01 启动时忽略用户设置的 TEXMACS_PATH，始终使用软件自带路径

### What
将 `src/System/Boot/init_texmacs.cpp` 中 `init_texmacs_path` 的逻辑从“仅在 `TEXMACS_PATH` 为空时兜底”改为“启动时直接设置程序自己的内置路径”。
这样 Linux、macOS 和 Windows 目标都会在进程初始化阶段覆盖用户环境里的 `TEXMACS_PATH`。

### Why
此前启动逻辑会先读取用户环境中的 `TEXMACS_PATH`，只有在它为空时才回退到程序自带路径。
一旦用户环境中存在错误值，插件系统和资源加载就会跟着读取错误目录，导致程序无法启动或加载错误资源。

### How
`src/System/Boot/init_texmacs.cpp` :161 使用 `builtin_texmacs_path` 替代 `current_texmacs_path`，不再依据用户环境决定是否设置 `TEXMACS_PATH`
`src/System/Boot/init_texmacs.cpp` :164 Linux 分支仅根据可执行文件附近的 `../share/<PREFIX_DIR>` 推导内置路径，并在存在时直接写入环境变量
`src/System/Boot/init_texmacs.cpp` :206 macOS 分支始终将 `TEXMACS_PATH` 设为 bundle 内的 `../Resources/share/<PREFIX_DIR>`
`src/System/Boot/init_texmacs.cpp` :219 Windows 分支始终将 `TEXMACS_PATH` 设为可执行文件上级目录推导出的内置路径
`src/System/Boot/init_texmacs.cpp` :242 最终校验改为读取 `resolved_texmacs_path`，校验的是进程内最终生效的路径，而不是已经废弃的用户输入值
