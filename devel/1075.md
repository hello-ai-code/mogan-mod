## 5 What
移除 `ChatConversationPanel::focusInput()` 中对 `set_current_view` 和 `call("update-menus")` 的调用，仅保留 Qt 层面的 `setFocus`。

## 6 Why
在 dock 模式下，LLM 生成时 `focusInput()` 调用 `set_current_view` 会将全局 `the_view` 切换到 chat input buffer 的 passive view（`win=NULL`），且不会恢复。这导致以下问题：

1. **导出 PDF 内容为空白**：`wrapped-print-to-file` 中 `(current-buffer)` 返回的是 chat buffer 而非主编辑器，`buffer-copy` 复制了 chat buffer 的内容。
2. **导出文件名异常**：文件名可能显示为 `chat-tab` 相关名称。
3. **打开 PDF 时 crash**：`load-pdf-buffer` → `switch-to-buffer` → `get_current_window()` → `ASSERT(vw->win != NULL)` 失败。

## 7 How
`focusInput()` 原本的目的是让用户聚焦到 chat 输入框。原实现通过 `set_current_view` 将全局 `the_view` 切到 chat input 的 passive view，再调用 `update-menus` 刷新菜单。

实际上这两步是多余的，且有严重副作用：

- **Qt focus 机制已足够**：`inputQTMWidget_->setFocus()` 会触发 `QTMWidget::focusInEvent` → `handle_keyboard_focus` → `focus_on_this_editor`，通过 Qt 事件队列异步完成 `set_current_view`，不会在 Scheme 执行中间打断。
- **`update-menus` 对 chat buffer 几乎无影响**：`update_menus()` 内部检测到 `is_chat_tab_buffer` 后，会跳过主菜单、主工具栏、focus 工具栏、侧边工具栏等绝大部分更新，仅更新 mode icons 和 tab pages。而 `focus_on_this_editor` 的异步调用也会触发 `update_menus`，无需在此手动调用。
- **副作用过大**：同步的 `set_current_view` 将 `the_view` 永久切到 `win=NULL` 的 passive view，破坏了主编辑器的 `current-buffer` 语义。

修改后 `focusInput()` 仅执行 Qt 层面的焦点切换，不影响全局 `the_view`。
