## 5 What

修复 LLM 聊天功能中的三个问题：

1. **停止会话时子进程残留**：点击停止按钮后，虽然 connection 被 stop，但 plugin 的 pending 任务未被清理，导致子进程可能残留。
2. **上下文角色错乱**：当 assistant 回复为空（或只有空白字符）时，`chat-tab-extract-rounds` 会将其错误地归类为 user 消息，导致构建的上下文中出现连续两个 user 消息，违反大多数 LLM API 的 role 交替要求。
3. **调试代码污染**：`chat-tab-extract-rounds` 和 `chat-tab-build-context` 中残留大量 `display` 调试输出，影响日志质量。

## 6 Why

- **子进程残留**：用户点击停止后，底层 HTTP 流式请求仍在进行，子进程未被正确杀死，可能导致资源泄漏或后续对话异常。
- **角色错乱**：在某些情况下（如网络错误、服务端返回空内容），assistant 节点可能为空。此时如果仍将该轮次视为 assistant，会导致 API 请求失败（如 400 Bad Request: roles must alternate）。如果视为 user，则会出现连续 user 消息。正确做法是在提取后合并连续相同 role 的消息。
- **调试输出**：这些 `display` 语句是在开发阶段加入的，现已不再需要，应清理以保持代码整洁。

## 7 How

### 7.1 修复子进程残留

将 `plugin-eval.scm` 中的 `plugin-cancel` 从普通 `define` 改为 `tm-define`，使其可以被外部模块调用：

```scheme
(tm-define (plugin-cancel lan ses dead?)
  ...)
```

然后在 `chat-adapter.scm` 的停止逻辑中，在 `(connection-stop ...)` 之后追加 `(plugin-cancel "llm" plugin-ses #t)`，确保 pending 任务被清空。

### 7.2 修复上下文角色错乱

新增 `chat-tab-merge-consecutive-rounds` 函数：

```scheme
(define (chat-tab-merge-consecutive-rounds rounds)
  (let loop ((items rounds) (acc '()))
    (if (null? items)
      (reverse acc)
      (let ((curr (car items)))
        (if (null? acc)
          (loop (cdr items) (cons curr acc))
          (let ((prev (car acc)))
            (if (string=? (car prev) (car curr))
              (loop (cdr items)
                (cons (cons (car curr)
                        (string-append (cdr prev) "\n" (cdr curr)))
                  (cdr acc)))
              (loop (cdr items) (cons curr acc)))))))))
```

该函数遍历 rounds 列表，当遇到连续相同 role 时，将它们的文本用换行符拼接合并。

同时，在 `chat-tab-extract-rounds` 中：
1. 增加对 assistant 文本的 `string-trim-spaces` 检查，如果 trim 后为 null，则将该轮视为 user（因为实际上没有 assistant 内容）
2. 在返回前调用 `chat-tab-merge-consecutive-rounds` 进行合并，确保最终不会出现连续相同 role

### 7.3 清理调试输出

删除 `chat-tab-extract-rounds` 和 `chat-tab-build-context` 中的所有 `display` 调试语句。
