## 5 What

1. `chat-tab-session-send` 增加延迟初始化：首次发送时检测 msg-buf 是否为空，若为空则执行 `session-enable-text-input` + `buffer-set-body '(session ...)` + `chat-tab-add-default-style-packages!`
2. 新增 `chat-tab-load-input-styles!`：仅对 input buffer 加载样式包
3. `ensureNewConversation` 调用 `chat-tab-load-input-styles!`，确保新对话输入框有正确样式

## 6 Why

0252 重构将旧的 `chat-tab-ensure-session!`（延迟初始化，发送时触发）替换为 `chat-tab-init-session!`（面板创建时立即调用）。但 `ensureNewConversation`（新对话路径）未调用 `chat-tab-init-session!`，而 `getOrCreatePanel`（恢复路径）调用了。

导致两个问题：
- 新对话的 input buffer 和 msg-buf 均未加载 `llm.ts` 等样式包
- 若在 `ensureNewConversation` 中直接调用 `chat-tab-init-session!`，会向 msg-buf 写入 `(session ...)` 结构，导致欢迎界面消失

## 7 How

### 延迟初始化恢复
旧代码通过 `chat-tab-ensure-session!` 在首次发送时初始化（`chat-tab-style-loaded` ahash-table 幂等保护）。新代码复用 `chat-tab-buffer-empty?` 作为幂等守卫：
- msg-buf 为空 → 执行初始化（设置 session body + 加载样式 + 注册 text-input）
- msg-buf 非空 → 跳过（已初始化或已恢复）

`chat-tab-buffer-empty?` 检查 `(document "")` 等空文档形态，语义上等价于"首次发送"。`session-enable-text-input` 本身是幂等的（ahash-set!），重复调用无副作用。

### input buffer 样式提前加载
新对话的 input buffer 需要从一开始就有正确的样式包（否则用户输入体验异常）。通过新增 `chat-tab-load-input-styles!` 在面板创建后单独加载，不涉及 msg-buf 操作。

### `add-style-package` 幂等性
`add-style-package` 内部调用 `set-style-list`，有 `(!= l (get-style-list))` 守卫且 `normalize-style-list` 去重，重复调用安全。
