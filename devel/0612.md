## 5 What

1. 将 TikZ 插件的图片生成流程从 `latex -> dvips -> gs -> png` 改为 `pdflatex -> pdf`
2. 利用 Mogan 内置的 MuPDF 直接渲染 PDF 图片，无需外部 Ghostscript
3. 移除 `init-tikz.scm` 中对 `(binary gs)` 的依赖
4. 移除 `tm-tikz.scm` 中的 `run-dvips`、`run-gs`、`png-size`、`eps-bbox-empty?` 函数
5. 添加 `image-valid?` 函数用于检测生成的 PDF 文件是否有效
6. 在 `binary/pdflatex.scm` 中添加 `find-binary-pdflatex` 查找逻辑
7. 更新单元测试，移除不再需要的 PNG/EPS 相关测试，保留核心逻辑测试

## 6 Why

Mogan STEM 使用 Goldfish Scheme 重写的 TikZ 插件最初依赖 Ghostscript 将 EPS 转为 PNG。这要求用户额外安装 Ghostscript，增加了环境配置负担。实际上 Mogan 内置了 MuPDF 渲染器，可以直接加载和显示 PDF 文件。因此，将插件改为直接生成 PDF 并交由 Mogan 内部渲染，可以彻底去掉 Ghostscript 依赖。

此外，原先为了处理空 bounding box 而引入的 `eps-bbox-empty?` 和 `png-size` 检测逻辑，本质上是 Ghostscript 拒绝处理极小页面时的 workaround。改用 MuPDF 后，即使页面极小也能正常渲染，因此这些 workaround 可以一并移除，代码更简洁。

## 7 How

### 7.1 改用 pdflatex 直接生成 PDF

在 `tm-tikz.scm` 中，将 `run-latex` + `run-dvips` + `run-gs` 三段式流程替换为单一的 `run-pdflatex`：

```scheme
(define (run-pdflatex tex-path pdflatex-bin)
  (let* ((inner-cmd (string-append (goldfish-quote pdflatex-bin)
                                   " --interaction=errorstopmode -halt-on-error "
                                   (goldfish-quote tex-path)
                                   " > /dev/null 2>&1"))
         (cmd (string-append "sh -c " (goldfish-quote inner-cmd)))
         (orig-dir (getcwd)))
    (unsetenv "DYLD_LIBRARY_PATH")
    (unsetenv "DYLD_FRAMEWORK_PATH")
    (unsetenv "DYLD_FALLBACK_LIBRARY_PATH")
    (unsetenv "DYLD_FALLBACK_FRAMEWORK_PATH")
    (chdir (tikz-temp-dir))
    (let ((result (os-call cmd)))
      (chdir orig-dir)
      result)))
```

`pdflatex` 直接在临时目录生成 `.pdf` 文件，参数与 `latex` 相同。

### 7.2 简化 flush-image

`flush-image` 不再需要区分 PNG/EPS，只需检查 PDF 文件是否存在且非空：

```scheme
(define (image-valid? path)
  (and (file-exists? path) (> (path-getsize path) 10)))

(define (flush-image path width height)
  (if (image-valid? path)
      (flush-file (string-append path "?" "width=" width "&" "height=" height))
      (flush-verbatim "Failed to generate image")))
```

### 7.3 更新 init-tikz.scm

移除 `(binary gs)` 模块和 `gs` 二进制参数，仅保留 `pdflatex`：


## 9 修复默认插入大小异常

### 9.1 问题现象

切换为 PDF 输出后，默认插入的 TikZ 图片大小比之前（PNG 输出）大太多。

### 9.2 根因分析

Mogan 的插件协议处理 `flush-file` 时，`file_flush` 函数在解析 URL 的 `width=0px&height=0px` 查询参数后，对 PNG 和 SVG 有原生尺寸回退：

- PNG：`native_image_size` 读取实际像素尺寸
- SVG：`svg_image_size` 读取实际尺寸

但对 PDF 没有类似回退。当 `width == 0 && height == 0` 时，`image_flush` 默认将尺寸设为 `0.618par`，导致 PDF 图片被拉伸到接近页面宽度的 61.8%，视觉上远大于原先的 PNG。

### 9.3 修复方案

在 `tm-tikz.scm` 的 `eval-and-print` 中，当用户未指定宽高（即 `width="0px"` 且 `height="0px"`）时，将默认宽度设为 `0.3par`，高度留空由 Mogan 按比例自适应：

```scheme
(let ((final-width width)
      (final-height height))
  (when (and (string=? width "0px") (string=? height "0px"))
    (set! final-width "0.3par")
    (set! final-height ""))
  (flush-image pdf-path final-width final-height))
```

- `0.3par` 约为页面宽度的 30%，视觉上与旧流程输出的 PNG 尺寸接近
- 不读取 pdflatex 的 `.log` 文件，避免文件解析带来的复杂性和潜在卡死风险
- 此方案完全在 TikZ 插件内部实现，不修改 Mogan 核心代码

## 10 修复 \usetikzlibrary 支持

### 10.1 问题现象

当 TikZ 会话中输入以 `\usetikzlibrary` 开头的代码时，例如：

```tikz
\usetikzlibrary{shapes.geometric}
\node[draw, circle] at (0,0) {A};
\node[draw, rectangle] at (2,0) {B};
\node[draw, diamond] at (1,1.5) {C};
\draw[->] (0,0) -- (2,0);
\draw[->] (2,0) -- (1,1.5);
```

pdflatex 编译报错，输出 `pdflatex error`。

### 10.2 根因分析

`wrap-tikz-code` 函数在处理以 `\usetikzlibrary` 开头的代码时，直接将整个代码块放入 `standalone` 文档的 `document` 环境中，没有为后续的绘图命令（如 `\node`、`\draw`）包裹 `tikzpicture` 环境。这导致 pdflatex 报错，因为这些命令必须在 `tikzpicture` 环境中才能执行。

旧逻辑生成的 LaTeX：

```latex
\documentclass[tikz]{standalone}
\begin{document}
\usetikzlibrary{shapes.geometric}
\node[draw, circle] at (0,0) {A};  % 报错：不在 tikzpicture 中
...
\end{document}
```

### 10.3 修复方案

修改 `wrap-tikz-code` 函数，将 `\usetikzlibrary{...}` 命令提取到 `document` 环境的顶层，然后把剩余的绘图代码包裹在 `tikzpicture` 环境中：

```scheme
(define (wrap-tikz-code code)
  (let ((trimmed (string-trim-left code)))
    (if (string-starts? trimmed "\\documentclass")
        code
        (let* ((lines (string-split code #\newline))
               (library-lines
                 (filter (lambda (line)
                           (string-starts? (string-trim-left line) "\\usetikzlibrary"))
                         lines))
               (other-lines
                 (filter (lambda (line)
                           (not (string-starts? (string-trim-left line) "\\usetikzlibrary")))
                         lines))
               (body (string-join other-lines "\n"))
               (body-trimmed (string-trim-left body)))
          (let ((inner-code
                  (if (or (string-null? body-trimmed)
                          (string-starts? body-trimmed "\\begin{tikzpicture}"))
                      body
                      (string-append "\\begin{tikzpicture}\n" body "\n\\end{tikzpicture}"))))
            (string-append
              "\\documentclass[tikz]{standalone}\n\\begin{document}\n"
              (if (null? library-lines) "" (string-append (string-join library-lines "\n") "\n"))
              inner-code
              "\n\\end{document}"))))))
```

修复后生成的 LaTeX：

```latex
\documentclass[tikz]{standalone}
\begin{document}
\usetikzlibrary{shapes.geometric}
\begin{tikzpicture}
\node[draw, circle] at (0,0) {A};
...
\end{tikzpicture}
\end{document}
```

### 10.4 测试覆盖

2. 多个 `\usetikzlibrary` -> 全部保留在 document 顶层

## 11 增加 TikZ 语法高亮功能

### 11.1 任务要求

增加 TikZ 插件的语法高亮和高级编辑支持，参考 MATLAB、Python 等已有插件的语法高亮实现方式。

### 11.2 解决思路与设计

在 Mogan / TeXmacs 中，对编程语言或会话语言（通过 `prog-language` 变量控制）的语法高亮、自动括号配对和缩进支持是通过一组约定的 Scheme 模块定义的：
1. C++ 端在构建特定语言的解析器时，会执行 `prog_language(string name)` 函数。该函数中有一个底层硬性过滤条件：若 `format_exists(name)` 为假（即格式不存在），语言解析器会直接降级为 `verb_language_rep`（即不支持任何高亮的纯文本模式），从而根本不会尝试去加载 `[name]-lang.scm`。因此，我们必须为 `tikz` 定义并注册其专属数据格式。
2. C++ 端成功验证格式存在后，会自动通过 Scheme 接口加载并运行 `(code [language-name]-lang)` 模块来解析 and 绘制高亮。
3. 编辑和模式相关的逻辑可以通过定义 `(code [language-name]-mode)` 与 `(code [language-name]-edit)` 模块实现，从而定制缩进格式、快捷键和括号的自动配对匹配。

因此，我们为 TikZ 插件量身定制了数据格式与以上三个代码层面的 Scheme 模块，使其语法高亮与流畅编辑开箱即用。

## 12 修复直线与点产生 0x0 bounding box 的空页面误判 bug

### 12.1 问题现象

在 TikZ 会话中绘制直线（如 `\draw (0,0) -- (0,1);`）或绘制孤立点（如 `\draw (0,0);`）时，系统提示：
`TikZ produced an empty image (0x0 bounding box)`
但实际上这是一个合法的绘制命令，不应判断为空。

### 12.2 根因分析

1. `pdf-page-empty?` 函数通过解析编译生成的 `.log` 文件中的 `papersize=w,h` 数据来判定页面是否为空。
2. 判定空页面的阈值原为 `1.0` pt。即若宽度 $w \le 1.0$ 且高度 $h \le 1.0$，则认为页面为空。
3. 旧逻辑使用的空判定条件是 `or` 运算：
   ```scheme
   (or (not w) (not h)
       (<= w 1.0) (<= h 1.0))
   ```
   这意味着，当绘制**水平直线**时（高度 $h \le 1.0$）或**垂直直线**时（宽度 $w \le 1.0$），都会触发 `or` 运算的分支，从而错误地将其标记为“空页面”，输出 `0x0 bounding box` 提示。
4. 真正合法的空页面检测应该是：**宽度与高度同时小于一定阈值**。因此应将 `or` 判定逻辑更正为 `and`。
5. 此外，当用户绘制孤立点（如 `\draw (0,0);`）时，由于 TikZ 会根据默认线宽（`thin` 默认为 0.4pt）进行裁剪，其编译产生的实际 `papersize` 大小约为 `0.4pt,0.4pt`。若空判定阈值设为原来的 `1.0` pt，则 `0.4 <= 1.0` 依然判定该孤立点为空页面。为了支持孤立点的正常渲染绘制，必须**将检测阈值进一步降低到 `0.1` pt**。

### 12.3 修复/实现方案

1. **重构并新增大小提取函数 `get-pdf-page-size`** (`TeXmacs/plugins/tikz/goldfish/tm-tikz.scm`)：
   - 从 `pdf-page-empty?` 中抽离出 `get-pdf-page-size`，专门用于从 pdflatex 的 `.log` 文件中解析出编译图像的真实 papersize 的 `(w, h)` 值。
   - 这不仅让空判定逻辑更为直观、模块化，更重要的是能在图片生成成功后，直接提取真实的物理宽高点数。

2. **更新空页面检测 `pdf-page-empty?` 并降低检测阈值至 `0.1` pt**：
   - 将宽和高同时 $\le 1.0$ 的检测硬编码阈值**降低至 $\le 0.1$ pt**。由于孤立点的 papersize 典型值为 $0.4$ pt（大于 $0.1$ pt），因此可以被成功识别为非空图案进行渲染；而真正空白无画面的文档输出大小为 $0.0$ pt，可以被完美过滤。
     ```scheme
     (define (pdf-page-empty? log-path)
       (let ((size (get-pdf-page-size log-path)))
         (if (not size)
             #t
             (let ((w (car size))
                   (h (cadr size)))
               (and (<= w 0.1) (<= h 0.1))))))
     ```

3. **修复直线的拉伸变形问题**：
   - 之前在 `eval-and-print` 中，当未手动指定尺寸时，默认会设 `width = 0.3par` 且 `height = 0px`（不限制）。
   - 这导致像垂直线 `\draw (0,0) -- (0,1);` 或水平线这样单维尺寸极小（如 0.4pt）的图像，在导入到 Mogan STEM 后，其极窄的宽度会被强行拉伸 300 多倍到 `0.3par`（约 150pt），在视觉上变成一个模糊扭曲、黑白渐变扩散的巨大正方形。
   - 为了实现像素级的精准无损渲染，我们将未指定宽高时的默认行为，修改为**自动获取 TikZ 编译后的真实 papersize（以 `pt` 点数为单位）**并直接作为图片的 `width` 和 `height` 传递给 `flush-image`：
     ```scheme
     (let ((final-width width)
           (final-height height))
       (when (and (string=? width "0px") (string=? height "0px"))
         (if size
             (begin
               (set! final-width (string-append (number->string (car size)) "pt"))
               (set! final-height (string-append (number->string (cadr size)) "pt")))
             (begin
               (set! final-width "0.3par")
               (set! final-height "0px"))))
       (flush-image pdf-path final-width final-height))
     ```
   - 如此一来，垂直线、水平线和孤立散点都会以其实际点数（如 0.4pt）的无损比例被精准渲染出来，彻底解决拉伸变形和渐变块的问题。

## 13 修复 extra_chars 格式及空格关键字的高亮失效 bug

### 13.1 问题现象

部分 TikZ 关键字（如库名 `shapes.geometric`、连线 `--`、箭头 `->`、选项 `line width` 等）无法被正确染色或在部分环境下失效。

### 13.2 根因分析

1. **`extra_chars` 的长度限制**：
   在 `tikz-lang.scm` 中，之前我们将 `extra_chars` 定义为单字符串 `"_.-"`。但在 Mogan C++ 后端 `prog_language.cpp:80` 中，对 `extra_chars` 的长度有硬性校验：`if (N(extra_char) == 1)`。由于 `"_.-"` 的长度是 3，导致它被整个忽略，从而点 `.` 和连字符 `-` 从未被注入到词法解析器中，导致所有包含它们的特殊关键字和操作符都无法被正确分词。
2. **空格关键字的匹配限制**：
   Mogan 的 C++ 词法解析器在匹配关键字时，不支持带空格的词条（如 `line width`, `inner sep`），而是会根据空格对其进行强行分词（变成 `line` 与 `width` 等）。

### 13.3 修复/实现方案

1. **更正 `extra_chars` 的定义格式** (`TeXmacs/plugins/tikz/progs/code/tikz-lang.scm`)：
   - 将其修正为独立字符列表，使其符合 C++ 端的逐字长判定要求：
     ```scheme
         (extra_chars "_" "." "-")
     ```
2. **重构拆分带空格的属性键名**：
   - 将所有带空格的复核键名（如 `line width`, `inner sep`, `rounded rectangle` 等）在 `tikz-lang.scm` 中拆分为单字关键字（如 `line`, `width`, `sep`, `inner`, `rounded`, `rectangle` 等）分别进行定义。这样当用户组合使用这些属性时，两个词都将被完美识别并高亮显示。

## 14 增加对 \usepackage 与 chemfig 的支持

### 14.1 问题现象

在 TikZ 交互式会话或会话文档中，如果输入 `\usepackage{chemfig}`，并使用 `\chemfig{*6(=-=-=-)}` 等命令，由于原有的 `wrap-tikz-code` 会将非 `\usetikzlibrary` 开头的行强行包裹在 `tikzpicture` 环境中，导致：
1. `\usepackage{...}` 出现在 `tikzpicture` 甚至是 `document` 环境中，从而引发 pdflatex 编译报错 `pdflatex error`。
2. `chemfig` 本身拥有独立的绘图逻辑（内部包含自己的 `tikzpicture`），将其嵌套包裹在 `tikzpicture` 中会导致生成的 PDF 格式/页面极度异常（变成多页且 papersize = 0.0pt,0.0pt），导致图片无法正常渲染。

### 14.2 根因分析

1. `wrap-tikz-code` 没有支持 `\usepackage`。所有以 `\usepackage` 开头的行都被分类为普通的绘图代码，从而留在 body 中。
2. 包裹逻辑：所有不以 `\usetikzlibrary` 或 `\begin{tikzpicture}` 开头的 body 都会被强行包裹在 `\begin{tikzpicture}...\end{tikzpicture}` 中。

### 14.3 修复方案

1. **提取 `\usepackage` 命令行到 Preamble**：
   在 `wrap-tikz-code` 中，提取所有以 `\usepackage` 开头的行（`package-lines`），并将它们放置在生成的 LaTeX 文档的 Preamble（即 `\documentclass[tikz]{standalone}` 之后，`\begin{document}` 之前）。
2. **解耦 `tikzpicture` 包裹逻辑与自动加载 `chemfig` 宏包**：
   在 `wrap-tikz-code` 中将“是否需要加载 `chemfig` 宏包”与“是否需要包裹 `tikzpicture`”这两个逻辑进行正交解耦，实现了完美的鲁棒性：
   - **宏包检测与自动加载** (`need-chemfig-package?`)：只要正文中**包含**任一 `\chem`（如嵌套在 `\node` 里的 `\chemfig`）或 `\scheme` 关键字，或者 `package-lines` 本身包含了 `chemfig`，我们就判定当前文档需要 `chemfig` 支持，进而在没有显式加载时自动在 Preamble 载入 `\usepackage{chemfig}`。
   - **`tikzpicture` 包裹绕过** (`has-chemfig?`)：只有当正文**最外层**以 `\chem` 或 `\scheme` 开头（即作为一个独立的分子、反应式渲染）时，我们才绕过第一层的 `tikzpicture` 环境包裹。而像嵌套在 `\node` 里的 `\chemfig`（最外层是以 `\node` 开头绘制复杂的 TikZ 图），则继续执行 `tikzpicture` 的自动包裹，从而使两者的混用天然和谐：
   ```scheme
   (let* ((has-chemfig? (or (string-starts? body-trimmed "\\chem")
                           (string-starts? body-trimmed "\\scheme")))
          (need-chemfig-package?
            (or has-chemfig?
                (string-contains? body "\\chem")
                (string-contains? body "\\scheme")))
          (extra-packages
            (if (and need-chemfig-package?
                     (null? (filter (lambda (line) (string-contains? line "chemfig"))
                                    package-lines)))
                '("\\usepackage{chemfig}")
                '()))
          (inner-code
            (if (or (string-null? body-trimmed)
                    (string-starts? body-trimmed "\\begin{tikzpicture}")
                    has-chemfig?)
                body
                (string-append "\\begin{tikzpicture}\n" body "\n\\end{tikzpicture}"))))
   ```
3. **保持测试覆盖与同步**：
   - 更新 `TeXmacs/plugins/tikz/goldfish/tm-tikz.scm` 中的 `wrap-tikz-code`。
   - 更新 `TeXmacs/plugins/tikz/tests/tm-tikz-test.scm` 中的 `wrap-tikz-code` 单元测试模拟函数。
   - 在 `tm-tikz-test.scm` 中增加对自动加载 `\usepackage{chemfig}`、显式声明及普通画笔混合的测试样例，确保 100% 通过。

## 15 增加对 \usepackage{optikz} 与 calc 的支持

### 15.1 问题现象

在 `0612.tmu` 最后一个节「光学图」中包含两个光学组件绘图测试用例：
1. 包含 `\usepackage{optikz}` 与 `\usepackage{calc}` 但未定义 `\skala` 变量，导致编译报错。
2. 包含 `\def\skala{0.8}` 等，但由于未使用 `rgb` 选项进行编译，导致 `drawrainbow` 绘图动作触发 unsupported color model `hsb` 报错。
3. 如果单独执行第二个用例，由于 `tm-tikz.scm` 是基于每个输入块独立编译，在无 `\usepackage{optikz}` 显式声明的情况下，编译报错无法识别光学元件指令（如 `\spectrometer` 等）。

### 15.2 根因分析

1. `\skala` 是用户自定义变量，首个用例遗漏了该变量的定义。
2. `optikz` 依赖 `hsb` 颜色模型（绘制彩虹激光束），但 `pgf` 默认未开启 `hsb`。需要在 `\documentclass[tikz,rgb]{standalone}` 中加入 `rgb` 选项，以便底层完美传递至 `xcolor` 并开启 `hsb` 颜色模型支持。
3. 交互会话中的后续输入块不共享内存中的 `\usepackage` 导入，因此必须对 `optikz` 的关键字（如 `\spectrometer`, `\camera`, `\diode` 等）进行类似 `chemfig` 的自动分析和按需加载。

### 15.3 修复方案

1. **升级 documentclass 模板**：
   在 `wrap-tikz-code` 中，将：
   `\documentclass[tikz]{standalone}`
   修改为：
   `\documentclass[tikz,rgb]{standalone}`
   以天然对 `optikz` 彩虹激光束的 `hsb` 颜色模式完美提供支持。

2. **支持 `optikz` 与 `calc` 关键字自动检测及加载**：
   通过 `need-optikz-package?` 分析绘图主体是否包含：`\spectrometer`, `\camera`, `\diode`, `\splitter`, `\convexlens`, `\concavelens`, `\planconvexlens`, `\mirror`, `\curvedmirror`, `\pockelscell`, `\laser`, `\grating`, `\drawrainbow`, `\TFP` 等光学核心符号，从而在需要时自动加载 `\usepackage{optikz}` 与 `\usepackage{calc}`。
